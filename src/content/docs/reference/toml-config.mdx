```toml
name = "my-api"
version = "1.0.0"

[database]
```

## Sections

### [project]

Project metadata and identification.

```toml
[project]
name = "my-api"           # Required: project name (alphanumeric, hyphens)
version = "1.0.0"         # Required: semantic version
description = "My API"    # Optional: project description
authors = ["you@email"]   # Optional: list of authors
```

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | Yes | Project identifier |
| `version` | string | Yes | Semantic version (x.y.z) |
| `description` | string | No | Human-readable description |
| `authors` | array | No | List of author emails |

### [database]

Database connection configuration.

```toml
[database]
type = "postgresql"
url = "${DATABASE_URL}"
pool_min = 5
pool_max = 20
statement_timeout = 30000
idle_timeout = 600000
ssl_mode = "require"
```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `type` | string | Required | Database type (`postgresql`) |
| `url` | string | Required | Connection URL or env var reference |
| `pool_min` | integer | `5` | Minimum pool connections |
| `pool_max` | integer | `20` | Maximum pool connections |
| `statement_timeout` | integer | `30000` | Query timeout in ms |
| `idle_timeout` | integer | `600000` | Idle connection timeout in ms |
| `ssl_mode` | string | `prefer` | SSL mode: `disable`, `prefer`, `require` |

**Environment variables:**

Use `${VAR_NAME}` syntax to reference environment variables:

```toml
url = "${DATABASE_URL}"  # Reads from DATABASE_URL env var
```

### [database.replica]

Read replica configuration for read/write splitting.

```toml
[database.replica]
url = "${DATABASE_REPLICA_URL}"
pool_min = 10
pool_max = 50
```

When configured, read queries (GraphQL queries) go to the replica, while write operations (mutations) go to the primary.

```toml
host = "0.0.0.0"
port = 8080
workers = 4
request_timeout = 30000

```

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `host` | string | `"0.0.0.0"` | Bind address |
| `port` | integer | `8080` | Listen port |
| `workers` | integer | CPU count | Worker threads |
| `request_timeout` | integer | `30000` | Request timeout in ms |
| `keep_alive` | integer | `75` | Keep-alive timeout in seconds |
| `max_request_size` | string | `"10mb"` | Max request body size |

| Field | Type | Default | Description |
|-------|------|---------|-------------|

| `origins` | array | `["*"]` | Allowed origins |
| `methods` | array | `["GET", "POST", "OPTIONS"]` | Allowed methods |
| `headers` | array | `["Content-Type"]` | Allowed headers |
| `credentials` | bool | `false` | Allow credentials |
| `max_age` | integer | `86400` | Preflight cache duration |

### [fraiseql.validation]

Comprehensive input validation configuration. Defines pre-execution validation rules for GraphQL input parameters at the **boundary** of your API—both for filtering operations (WHERE inputs) and data mutations (CREATE/UPDATE operations).

#### Overview

The validation system validates **all GraphQL input variables before execution**, ensuring data integrity across your entire API:

- **WHERE input validation**: Query parameters for filtering
- **Mutation input validation**: Arguments to CREATE/UPDATE operations
- **Field-level validation**: Applied universally to all input fields

All validation happens **before SQL execution**, catching errors fast without database overhead:

```toml
[fraiseql.validation]
# Validate email domains
email_domain_eq = { pattern = "^[a-z0-9]([a-z0-9-]*\\.)*[a-z0-9]([a-z0-9-]*[a-z0-9])?$" }

# Validate IBAN country codes
iban_country_eq = { checksum = "mod97" }

# Validate distance is reasonable
distance_within_radius_km = { numeric_range = { min: 0, max: 40075 } }

# Validate country codes
country_eq = { enum = ["US", "CA", "UK", "FR", "DE", "IT", "ES", ...] }
```

#### Validation Rule Types

FraiseQL supports 7 validation rule types:

##### Pattern (Regex)

Validate using regular expression:

```toml
[fraiseql.validation]
# Email domain format
email_domain_eq = { pattern = "^[a-z0-9]([a-z0-9-]*\\.)*[a-z0-9]([a-z0-9-]*[a-z0-9])?$" }

# API key format (alphanumeric, hyphens)
api_key_prefix_eq = { pattern = "^[a-zA-Z0-9-]{3,}$" }
```

##### Length

Exact length validation:

```toml
[fraiseql.validation]
# Country codes are always 2 characters
country_code = { length = 2 }
```

##### LengthRange

Minimum and maximum length:

```toml
[fraiseql.validation]
# Slugs must be 3-50 characters
slug = { length_range = { min = 3, max = 50 } }

# API keys are typically 32-64 characters
api_key = { length_range = { min = 32, max = 64 } }
```

##### Checksum

Algorithm-based validation (Mod97 for IBAN, Luhn for VIN/Credit Cards):

```toml
[fraiseql.validation]
# IBAN uses MOD-97 checksum
iban_country_eq = { checksum = "mod97" }

# VIN uses Luhn algorithm
vin_wmi_eq = { checksum = "luhn" }
```

##### NumericRange

Numeric value range:

```toml
[fraiseql.validation]
# Distance must be positive and less than Earth's circumference
distance_within_radius_km = { numeric_range = { min = 0, max = 40075 } }

# Percentage must be 0-100
discount_percentage = { numeric_range = { min = 0, max = 100 } }
```

##### Enum

Value must be one of allowed options:

```toml
[fraiseql.validation]
# Phone type must be one of these values
phone_type_eq = { enum = ["mobile", "fixed", "tollfree", "premium", "shared", "voip"] }

# Payment method restricted list
payment_method = { enum = ["credit_card", "bank_transfer", "paypal", "stripe"] }
```

##### All (Composite)

Multiple rules must all pass:

```toml
[fraiseql.validation]
# VIN WMI must be exactly 3 chars AND all uppercase
vin_wmi_eq = { all = [
    { length = 3 },
    { pattern = "^[A-Z0-9]{3}$" }
] }
```

#### Default Rules

FraiseQL includes 40+ default validation rules for common semantic types. Override defaults in your configuration:

```toml
[fraiseql.validation]
# Override the default email domain pattern
email_domain_eq = { pattern = "^custom-pattern$" }

# Add custom validation not in defaults
my_custom_field = { length_range = { min = 1, max = 255 } }
```

**Common Default Rules**:

| Operator | Default Rule | Purpose |
|----------|--------------|---------|
| `email_domain_eq` | Pattern (RFC domain) | Validate email domain format |
| `email_domain_in` | Pattern (RFC domain) | Validate multiple domains |
| `phone_country_code_eq` | Pattern (E.164) | Validate country code format |
| `coordinates_distance_within_radius_km` | NumericRange (0-40075) | Distance in km must be positive and realistic |
| `iban_country_eq` | Checksum (mod97) | IBAN checksum validation |
| `vin_wmi_eq` | Length (3), Pattern | World Manufacturer ID exact format |
| `cidr_contains_address` | Pattern (CIDR notation) | Valid CIDR block format |
| `country_code_eq` | Enum (ISO 3166) | Valid ISO 3166-1 country code |
| `currency_code_eq` | Enum (ISO 4217) | Valid ISO 4217 currency code |
| `timezone_eq` | Enum (IANA TZ) | Valid IANA timezone |

#### Custom Validation Examples

**Email Domain Allowlist**:

```toml
[fraiseql.validation]
# Only allow specific company domains
email_domain_eq = { enum = ["company.com", "company.co.uk", "subsidiary.com"] }
```

**API Key Format**:

```toml
[fraiseql.validation]
# API keys must start with 'sk_' and be 32+ characters
api_key_eq = { all = [
    { pattern = "^sk_[a-zA-Z0-9]+$" },
    { length_range = { min = 32, max = 64 } }
] }
```

**Geographic Bounds**:

```toml
[fraiseql.validation]
# Only allow searches within continental US (roughly)
latitude_eq = { numeric_range = { min = 25, max = 49 } }
longitude_eq = { numeric_range = { min = -125, max = -66 } }
distance_within_radius_km = { numeric_range = { min = 0.1, max = 1000 } }
```

**Date Range Constraints**:

```toml
[fraiseql.validation]
# Projects must be at least 7 days
timeline_duration_gte = { numeric_range = { min = 7, max = 3650 } }
```

**Mutation Input Validation**:

The same validation rules apply to mutation inputs. For example, when creating a user via mutation:

```toml
[fraiseql.validation]
# Validate email domains on create/update
email_domain_eq = { pattern = "^[a-z0-9]([a-z0-9-]*\\.)*[a-z0-9]([a-z0-9-]*[a-z0-9])?$" }

# Country code must be valid
country_code = { enum = ["US", "CA", "UK", "FR", "DE", "IT", "ES"] }

# Age must be realistic (for user registration)
age = { numeric_range = { min = 1, max = 150 } }
```

Both WHERE input parameters and mutation arguments are validated using the same rules:

```graphql
# Query: email domain validated in WHERE clause
query {
  users(where: { email: { domainEq: "company.com" } }) {
    id email
  }
}

# Mutation: email domain validated in input
mutation {
  createUser(input: {
    name: "Alice"
    email: "alice@company.com"    # Validated by email_domain_eq
    country: "US"                 # Validated by country_code enum
    age: 28                        # Validated by age numeric_range
  }) {
    id email
  }
}
```

#### Integration with Rich Filters

Validation rules work seamlessly with [Rich Filters](/features/rich-filters) for WHERE inputs:

```graphql
query {
  users(where: {
    email: { domainEq: "example.com" }  # Validated via fraiseql.validation
    location: { distanceWithin: { latitude: 40.7, longitude: -74.0, radiusKm: 5 } }
  }) {
    id
    email
  }
}
```

#### Integration with Mutations

Validation also applies to mutation inputs, ensuring data quality on writes:

```graphql
mutation {
  # All input fields validated before execution
  createUser(input: {
    name: "Alice"              # Validated against name rules
    email: "alice@example.com"  # Validated against email rules
    country: "US"              # Validated against country enum
  }) {
    id
    email
  }
}
```

The same validation rules apply whether parameters come from **query WHERE inputs** or **mutation arguments**.

Validation happens **before** SQL execution, catching errors fast without database overhead.

#### Best Practices

1. **Validate at the boundary**: Use validation for all user-provided filter parameters
2. **Whitelist, don't blacklist**: Use `enum` for restricted lists rather than excluding values
3. **Geographic realism**: Set reasonable bounds on coordinates and distances
4. **Reference data sync**: Keep enum lists in sync with your database lookups

| Field | Type | Default | Description |
