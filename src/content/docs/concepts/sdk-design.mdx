FraiseQL SDKs are **compile-time schema authoring tools**, not runtime libraries. This page explains what that means and why it matters.

## The compile→run model

FraiseQL separates schema authoring (any language) from request handling (Rust):

```
SDK code → fraiseql compile → schema.json → fraiseql run → Rust server
```

1. **Author**: Define types, queries, and mutations in your language (Python decorators, TypeScript decorators, Go struct tags, Java annotations, etc.)
2. **Compile**: `fraiseql compile` executes your SDK code, collects the schema registry, and writes `schema.json`
3. **Run**: `fraiseql run` starts the Rust server using `schema.json` and your database. The SDK code is never loaded again.

The SDK process runs once at build time, produces a JSON artifact, and exits. The Rust server has no FFI, no language runtime, no interpreter embedded. This is why FraiseQL supports 11 languages without 11 server implementations.

## What SDKs actually do

Every SDK provides:

- **Type definitions** — map language constructs (classes, structs, records) to GraphQL types
- **Query wiring** — connect a query name to a SQL view (`v_*`)
- **Mutation wiring** — connect a mutation name to a SQL function (`fn_*`)
- **Schema registry** — collect all definitions and export `schema.json`

Some SDKs also provide:

- **HTTP clients** — for consuming FraiseQL APIs (Ruby, Dart, C#, F#)
- **AI integrations** — OpenAI tool wrapping (Ruby), Gemini tool wrapping (Dart), Candle ML (Rust)
- **Transport annotations** — REST path/method decorators (Python, TypeScript, Go, Java, C#)

## What SDKs do NOT do

- Handle HTTP requests
- Execute GraphQL queries
- Connect to databases
- Run at application startup
- Add runtime overhead

<Aside type="note">
SDK decorators, annotations, and struct tags are metadata declarations. They register information in a schema registry. Function bodies (where they exist) are configuration calls — `fraiseql.config(sql_source="v_post")` — not executable business logic.
</Aside>

## Why this design?

**Performance**: The server is always compiled Rust. No interpreter overhead, no GC pauses, no cold starts from language runtimes.

**Language freedom**: Teams pick the language they know for schema authoring. The runtime characteristics are identical regardless of SDK choice.

**Simplicity**: The JSON schema is the contract between authoring and serving. You can inspect it, version it, diff it, and validate it independently.

**Security**: The attack surface is the Rust binary + PostgreSQL. SDK code never runs in production.

## The schema.json artifact

`fraiseql compile` produces a single `schema.json` file containing:

- GraphQL type definitions
- Query→view mappings with argument metadata
- Mutation→function mappings with operation types
- Transport annotations (REST paths, gRPC service names)
- Validation rules and authorization requirements

This file is the single source of truth for `fraiseql run`. You can compile on a developer machine and deploy only the JSON + the Rust binary.
