FraiseQL configuration lives in a single `fraiseql.toml` file. Different sections are consumed at different points in the build / boot lifecycle:

- The **`fraiseql-server` binary** reads `ServerConfig` directly from `fraiseql.toml` at startup. The sections it auto-wires are `[auth]`, `[auth_hs256]`, `[auth.me]`, `[server]`, `[server.cors]`, `[server.tls]`, `[tls]`, `[database]`, plus a handful of operator-facing knobs (`metrics_token`, `admin_token`, `admin_readonly_token`, `storage_token`).
- The **`fraiseql-cli compile` step** reads `fraiseql.toml` as a `TomlProjectConfig`, picks up `[security.*]` (and any subordinate blocks under it), and bakes them into `schema.compiled.json` under `schema.security.additional["<subsystem>"]`. The server reads from there at boot for: API keys, token revocation, PKCE, state encryption, error sanitization, per-endpoint rate limiting, and trusted documents.
- The **CLI alone** validates the security configuration with `serde(deny_unknown_fields)`. Editing `[security.*]` without re-running `fraiseql-cli compile` is a silent no-op at the runtime; the server only sees what compile bakes in.

The authoritative source for the full auth-and-security surface is [Building → Authentication](/building/authentication). This page is the map.

## Minimal configuration

```toml title="fraiseql.toml"
[project]
name = "my-api"
version = "1.0.0"

[database]
url = "${DATABASE_URL}"
```

Every other section has defaults. The `[fraiseql]` block becomes mandatory once you have a schema file (`schema_file` / `output_file` paths); otherwise the CLI uses `schema.json` / `schema.compiled.json` defaults.

## Configuration surface map

### Direct-from-ServerConfig (binary reads at startup; no compile step)

These sections live on `ServerConfig` and are auto-wired by `fraiseql-server` when present in `fraiseql.toml`. They are visible without `fraiseql-cli compile`.

| Section | What it configures | Authoritative reference |
|---------|-------------------|--------------------------|
| `[auth]` | OIDC token validator (issuer, audience, JWKS, algorithm allowlist) — production auth path | [Authentication → JWT and OIDC](/building/authentication#jwt-and-oidc) |
| `[auth.me]` | `GET /auth/me` session-identity endpoint — opt-in, rides on `[auth]` | [Authentication → `/auth/me`](/building/authentication#authme-session-identity-endpoint) |
| `[auth_hs256]` | HS256 symmetric-key validator — integration testing only, mutually exclusive with `[auth]` | [Authentication → HS256 testing mode](/building/authentication#hs256-testing-mode) |
| `[server]` | Bind address, port, request timeout, keep-alive, admin-token guards | [Reference → TOML config](/reference/toml-config) |
| `[server.cors]` | CORS origins, credentials, allowed methods | [Reference → TOML config](/reference/toml-config) |
| `[server.tls]` / `[tls]` | TLS certificate, private key, mTLS settings | [Reference → TOML config](/reference/toml-config) |
| `[database]` | Connection URL, pool sizing, SSL mode, connection / idle timeouts | [Reference → TOML config](/reference/toml-config) |

### Compile-step indirection (`[security.*]` — requires `fraiseql-cli compile`)

These sections live on `TomlProjectConfig.fraiseql.security` in the CLI's schema. The CLI validates them with `deny_unknown_fields`, bakes them into the compiled schema, and the server reads them from `schema.compiled.json` at boot. A TOML edit alone is a silent no-op — re-run compile after every change.

| Section | What it configures | Authoritative reference |
|---------|-------------------|--------------------------|
| `[security.api_keys]` | Static API keys with SHA-256 hashed storage | [Authentication → API keys](/building/authentication#api-keys) |
| `[security.token_revocation]` | JWT `jti` revocation (memory or Redis; postgres downgrades — FW-25) | [Authentication → Token revocation](/building/authentication#token-revocation) |
| `[security.pkce]` | PKCE OAuth2 challenge (S256 default; redis-pkce for multi-replica) | [Authentication → PKCE OAuth2 flow](/building/authentication#pkce-oauth2-flow) |
| `[security.state_encryption]` | AEAD encryption of OAuth state and PKCE blobs (ChaCha20-Poly1305 or AES-256-GCM) | [Encryption](/features/encryption) |
| `[security.rate_limiting]` | Per-endpoint rate caps for auth routes (failed-login fields silently dropped — FW-24) | [Rate limiting](/features/rate-limiting) |
| `[security.error_sanitization]` | Strip stack traces and DB errors from client responses (disabled by default) | [Security → error sanitization](/features/security#quick-mention--error-sanitization-query-complexity-cors) |
| `[security.trusted_documents]` | Query allowlist (strict or permissive mode) | [Reference → TOML config](/reference/toml-config) |
| `[security.enterprise]` | Audit logging flags (`audit_logging_enabled`, `audit_log_backend`) | [Audit logging](/features/audit-logging) |
| `[[security.rules]]` / `[[security.policies]]` / `[[security.field_auth]]` | Elo-rule and RBAC/ABAC policy declarations bound to types and fields | [Security → authorization policies](/features/security#authorization-policies-toml-driven) |
| `[fraiseql.tenancy]` | Tenant-mode discriminator (shared-pool / pool-per-tenant / schema-per-tenant) | [Multi-tenancy](/building/multi-tenancy) |
| `[auth]` (at the CLI's TomlSchema level) | OAuth2 client identity for the PKCE flow (discovery URL, client ID, client secret env var, server redirect URI) — distinct from the server's `[auth]` OIDC validator | [Authentication → PKCE OAuth2 flow](/building/authentication#pkce-oauth2-flow) |

## Example: production OIDC + compile-step subsystems

The shapes below match the `building/authentication` page exactly. Edit, then re-run `fraiseql-cli compile` before restarting the server.

```toml title="fraiseql.toml — production OIDC + PKCE + revocation"
[project]
name = "my-api"
version = "1.0.0"

[database]
url = "${DATABASE_URL}"
pool_min = 5
pool_max = 20
ssl_mode = "require"

[server]
host = "0.0.0.0"
port = 8080
admin_api_enabled = true
admin_token = "${FRAISEQL_ADMIN_TOKEN}"

[server.cors]
origins = ["https://app.example.com"]
credentials = true

# ---- Direct-TOML on ServerConfig (no compile step required) ----
[auth]                                     # OIDC validator — direct-wired
issuer = "https://your-tenant.auth0.com/"
audience = "https://api.example.com"       # SECURITY CRITICAL — boot fails without it

[auth.me]                                  # opt-in; rides on [auth]
enabled = true
expose_claims = ["tenant_id", "https://myapp.com/role"]

# ---- Compile-step indirection (re-run `fraiseql-cli compile` after edits) ----
[security.state_encryption]
enabled = true
algorithm = "chacha20-poly1305"
key_source = "env"
key_env = "STATE_ENCRYPTION_KEY"

[security.pkce]
enabled = true
code_challenge_method = "S256"
state_ttl_secs = 600

[security.token_revocation]
enabled = true
backend = "redis"                          # "postgres" silently downgrades — FW-25 #357
redis_url = "redis://redis:6379"
require_jti = true
fail_open = false

[security.error_sanitization]
enabled = true
hide_implementation_details = true
sanitize_database_errors = true
custom_error_message = "An internal error occurred"

[security.rate_limiting]
enabled = true
trust_proxy_headers = false                # requires trusted_proxy_cidrs if true
```

{/* source: crates/fraiseql-server/src/server_config/mod.rs:L40 — ServerConfig struct (no deny_unknown_fields, so unknown sections are silently ignored at server boot) */}

The server silently ignores `[fraiseql.*]` at boot — those keys are not present on `ServerConfig`. The CLI's compile step rejects unknown fields at the `[security.*]` level, so misspellings there fail loudly. Misspellings at the top level (`[fraseql.security.*]`, `[security.api_keys]` instead of `[security.api_keys]`) are NOT caught by the server — the keys end up parsed as unrelated top-level sections and silently dropped.

## Environment variable substitution

`fraiseql.toml` supports `${VAR_NAME}` substitution at parse time. Use it for secrets and per-environment values:

```toml
[database]
url = "${DATABASE_URL}"

[security.token_revocation]
redis_url = "${REDIS_URL}"

[server]
admin_token = "${FRAISEQL_ADMIN_TOKEN}"
```

{/* source: crates/fraiseql-cli/src/config/mod.rs:L1-L15 — expand_env_vars import + module documentation */}

### Variables with no TOML equivalent

These settings exist only as environment variables:

| Variable | Description |
|----------|-------------|
| `DATABASE_URL` | Database connection string (when `[database] url = "${DATABASE_URL}"`) |
| `RUST_LOG` | Log level — `error` \| `warn` \| `info` \| `debug` \| `trace` |
| `FRAISEQL_OBSERVERS_ALLOW_INSECURE` | Observer SSRF / scheme-validation bypass — never set in production. See [Observers security caveats](/features/observers#security-caveats). |
| `STATE_ENCRYPTION_KEY` | 32-byte secret consumed by `[security.state_encryption] key_source = "env"` |
| `OIDC_CLIENT_SECRET` | OAuth2 client secret for the PKCE flow; referenced from `[auth] client_secret_env` (CLI-level) |
| `FRAISEQL_HS256_SECRET` | HS256 shared secret; referenced from `[auth_hs256] secret_env` |
| `FRAISEQL_ADMIN_TOKEN` | Bearer token for the RBAC management API and other admin operations |
| `OTEL_EXPORTER_OTLP_ENDPOINT` | Trace export collector endpoint (fallback; prefer `[tracing] otlp_endpoint`) |

## Configuration by environment

Use separate `fraiseql.toml` files per environment, or vary values via `${VAR}` substitution:

```bash
# Development
DATABASE_URL=postgresql://localhost:5432/myapp_dev fraiseql-server --config fraiseql.toml

# Production (different .env file or secrets manager)
DATABASE_URL=postgresql://prod-host:5432/myapp fraiseql-server --config fraiseql.toml
```

You can also maintain separate config files:

```bash
fraiseql-cli compile --config fraiseql.production.toml --output schema.compiled.json
fraiseql-server --config fraiseql.production.toml
```

## Validation

The CLI's `fraiseql-cli compile` validates `fraiseql.toml` against the strict `TomlProjectConfig` schema. Unknown fields inside `[fraiseql.*]` blocks fail loudly:

```bash
fraiseql-cli compile
# Error: unknown field `unknown_field` at fraiseql.toml:N:N
```

The server's `--config` loader does NOT use `deny_unknown_fields` — unknown top-level sections (typos like `[secuirty]`, or sections the server does not know about) are silently ignored. If a section's behaviour depends on the compile step, run `fraiseql-cli compile` after every TOML edit and treat the compile error as the source of truth.

## Common patterns

### Local development (HS256 + no PKCE)

```toml title="fraiseql.toml"
[project]
name = "my-api"

[database]
url = "postgresql://localhost:5432/myapp_dev"

[server]
port = 8080

[auth_hs256]
secret_env = "FRAISEQL_HS256_SECRET"
issuer = "local-dev"
audience = "my-api"                         # operator-required; framework does not enforce (FW-27)
```

```bash
export FRAISEQL_HS256_SECRET="$(openssl rand -base64 32)"
RUST_LOG=debug fraiseql-server --config fraiseql.toml
```

### Production (OIDC + PKCE + revocation + state encryption)

See the full example earlier on this page. The four-line compile-step block:

```bash
export STATE_ENCRYPTION_KEY="$(openssl rand -hex 32)"
export FRAISEQL_ADMIN_TOKEN="$(openssl rand -hex 32)"
fraiseql-cli compile --output schema.compiled.json
fraiseql-server --config fraiseql.toml
```

### Multi-tenant (JWT-claim tenant dispatch)

```toml title="fraiseql.toml"
[project]
name = "multi-tenant-api"

[database]
url = "${DATABASE_URL}"

[auth]
issuer = "${OIDC_ISSUER}"
audience = "${OIDC_AUDIENCE}"

[fraiseql.tenancy]
mode = "shared_pool"                        # see Multi-tenancy page for "pool_per_tenant" / "schema_per_tenant"
discriminator_claim = "org_id"              # JWT claim that picks the tenant
```

See the [Multi-tenancy](/building/multi-tenancy) page for the tenant-registry / executor-factory wiring and the library-API caveat (FW-3 #330).

### Multi-transport (REST + gRPC + GraphQL on one port)

```toml title="fraiseql.toml"
[project]
name = "my-api"

[database]
url = "${DATABASE_URL}"

[server]
port = 8080

[rest]
path = "/rest/v1"
require_auth = false
openapi_enabled = true

[grpc]
enabled = true
```

REST endpoints exist for operations that include `rest_path` / `rest_method` annotations; gRPC services are auto-generated for all operations when `[grpc] enabled = true`. See [gRPC transport](/features/grpc-transport) and [REST endpoints](/features/rest-transport).
