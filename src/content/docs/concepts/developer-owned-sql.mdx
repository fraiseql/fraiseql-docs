FraiseQL is database-first. You write the SQL views that power your GraphQL API. This is a deliberate choice — and it's the reason FraiseQL can guarantee single-query execution, zero N+1 problems, and predictable performance.

## Why SQL, Not Abstraction

Most GraphQL frameworks generate SQL behind the scenes. You write resolvers or schema definitions, and something you can't see produces queries you can't easily debug. When performance degrades, you're reverse-engineering an abstraction layer.

FraiseQL takes the opposite approach: **you write the SQL**. The view you write is the query that runs. There's nothing hidden.

This gives you:

- **Predictable performance** — you see the query plan, you understand the cost
- **Full database power** — CTEs, window functions, stored procedures, custom aggregations (PostgreSQL, SQL Server, MySQL, SQLite — extensible to any database with a Rust driver)
- **No ORM impedance mismatch** — your data model is your data model, not a compromise
- **Debuggable queries** — `EXPLAIN ANALYZE` on the exact SQL that serves your API

## The Pattern Is Simple

Every SQL view in FraiseQL follows the same pattern: an `id` column and a `data` JSONB column.

<Tabs syncKey="db">
  <TabItem label="PostgreSQL">
    ```sql
    CREATE VIEW v_user AS
    SELECT
        u.id,
        jsonb_build_object(
            'id', u.id::text,
            'name', u.name,
            'email', u.email
        ) AS data
    FROM tb_user u;
    ```
  </TabItem>

  <TabItem label="MySQL">
    ```sql
    CREATE VIEW v_user AS
    SELECT
        u.id,
        JSON_OBJECT(
            'id', u.id,
            'name', u.name,
            'email', u.email
        ) AS data
    FROM tb_user u;
    ```
  </TabItem>

  <TabItem label="SQLite">
    ```sql
    CREATE VIEW v_user AS
    SELECT
        u.id,
        json_object(
            'id', u.id,
            'name', u.name,
            'email', u.email
        ) AS data
    FROM tb_user u;
    ```
  </TabItem>

  <TabItem label="SQL Server">
    ```sql
    CREATE VIEW dbo.v_user
    WITH SCHEMABINDING AS
    SELECT
        u.id,
        (
            SELECT u.id, u.name, u.email
            FOR JSON PATH, WITHOUT_ARRAY_WRAPPER
        ) AS data
    FROM dbo.tb_user u;
    ```
  </TabItem>
</Tabs>

That's it. The pattern is consistent across all databases, and you have a view that FraiseQL maps to a `User` GraphQL type.

### Composing Views

To embed related data, reference another view's `.data` column:

<Tabs syncKey="db">
  <TabItem label="PostgreSQL">
    ```sql
    CREATE VIEW v_post AS
    SELECT
        p.id,
        jsonb_build_object(
            'id', p.id::text,
            'title', p.title,
            'content', p.content,
            'author', vu.data
        ) AS data
    FROM tb_post p
    JOIN tb_user u ON u.pk_user = p.fk_user
    JOIN v_user vu ON vu.id = u.id;
    ```
  </TabItem>

  <TabItem label="MySQL">
    ```sql

            'content', p.content,
            'author', vu.data
        ) AS data
    FROM tb_post p
    JOIN tb_user u ON u.pk_user = p.fk_user
    JOIN v_user vu ON vu.id = u.id;
    ```
  </TabItem>

  <TabItem label="SQLite">
    ```sql
    CREATE VIEW v_post AS
    SELECT
        p.id,
        json_object(
            'id', p.id,
            'title', p.title,
            'content', p.content,
            'author', vu.data
        ) AS data
    FROM tb_post p
    JOIN tb_user u ON u.pk_user = p.fk_user
    JOIN v_user vu ON vu.id = u.id;
    ```
  </TabItem>

  <TabItem label="SQL Server">
    ```sql
    CREATE VIEW dbo.v_post
    WITH SCHEMABINDING AS
    SELECT
        p.id,
        (
            SELECT p.id, p.title, p.content, vu.data AS author
            FOR JSON PATH, WITHOUT_ARRAY_WRAPPER
        ) AS data
    FROM dbo.tb_post p
    JOIN dbo.tb_user u ON u.pk_user = p.fk_user
    JOIN dbo.v_user vu ON vu.id = u.id;
    ```
  </TabItem>
</Tabs>

`SELECT data FROM v_post` returns the full nested response — author included. One query. No N+1.

### One-to-Many With Aggregation

<Tabs syncKey="db">
  <TabItem label="PostgreSQL">
    ```sql
    CREATE VIEW v_user_with_posts AS
    SELECT
        u.id,
        jsonb_build_object(
            'id', u.id::text,
            'name', u.name,
            'posts', COALESCE(
                jsonb_agg(vp.data ORDER BY vp.data->>'created_at' DESC)
                FILTER (WHERE vp.id IS NOT NULL),
                '[]'::jsonb
            )
        ) AS data
    FROM tb_user u
    LEFT JOIN tb_post p ON p.fk_user = u.pk_user
    LEFT JOIN v_post vp ON vp.id = p.id
    GROUP BY u.id, u.name;
    ```
  </TabItem>

  <TabItem label="MySQL">
    ```sql
    CREATE VIEW v_user_with_posts AS
    SELECT
        u.id,
        JSON_OBJECT(
            'id', u.id,
            'name', u.name,
            'posts', COALESCE(
                JSON_ARRAYAGG(vp.data),
                JSON_ARRAY()
            )
        ) AS data
    FROM tb_user u
    LEFT JOIN tb_post p ON p.fk_user = u.pk_user
    LEFT JOIN v_post vp ON vp.id = p.id
    GROUP BY u.id, u.name;
    ```
  </TabItem>

  <TabItem label="SQLite">
    ```sql
    CREATE VIEW v_user_with_posts AS
    SELECT
        u.id,
        json_object(
            'id', u.id,
            'name', u.name,
            'posts', COALESCE(
                json_group_array(vp.data),
                json_array()
            )
        ) AS data
    FROM tb_user u
    LEFT JOIN tb_post p ON p.fk_user = u.pk_user
    LEFT JOIN v_post vp ON vp.id = p.id
    GROUP BY u.id, u.name;
    ```
  </TabItem>

  <TabItem label="SQL Server">
    ```sql
    CREATE VIEW dbo.v_user_with_posts
    WITH SCHEMABINDING AS
    SELECT
        u.id,
        (
            SELECT u.id, u.name,
                (
                    SELECT json_group_array(vp.data)
                    FROM dbo.tb_post p
                    JOIN dbo.v_post vp ON vp.id = p.id
                    WHERE p.fk_user = u.pk_user
                ) AS posts
            FOR JSON PATH, WITHOUT_ARRAY_WRAPPER
        ) AS data
    FROM dbo.tb_user u;
    ```
  </TabItem>
</Tabs>

The pattern is consistent: JSON object building for single objects, array aggregation for lists.

## AI-Friendly by Design

The view pattern is simple and consistent enough that language models generate it accurately — including local models like Llama and Mistral.

### Why It Works for LLMs

- **Consistent structure** — every view follows the same `id` + `data` pattern
- **Limited vocabulary** — `jsonb_build_object`, `jsonb_agg`, `COALESCE`, `FILTER`
- **Clear inputs** — the table schema defines what goes into the view
- **No ambiguity** — the naming convention (`tb_*`, `v_*`, `pk_*`, `fk_*`) removes guesswork

### Full-Text Search

<Tabs syncKey="db">
  <TabItem label="PostgreSQL">
    ```sql
    CREATE VIEW v_post_search AS
    SELECT
        p.id,
        jsonb_build_object(
            'id', p.id::text,
            'title', p.title,
            'snippet', ts_headline('english', p.content, query),
            'rank', ts_rank(p.search_vector, query)
        ) AS data
    FROM tb_post p, plainto_tsquery('english', $1) query
    WHERE p.search_vector @@ query;
    ```
  </TabItem>

  <TabItem label="MySQL">
    ```sql
    CREATE VIEW v_post_search AS
    SELECT
        p.id,
        JSON_OBJECT(
            'id', p.id,
            'title', p.title,
            'snippet', SUBSTRING(p.content, 1, 100),
            'rank', MATCH(p.title, p.content) AGAINST (? IN BOOLEAN MODE)
        ) AS data
    FROM tb_post p
    WHERE MATCH(p.title, p.content) AGAINST (? IN BOOLEAN MODE);
    ```
  </TabItem>

  <TabItem label="SQLite">
    ```sql
    CREATE VIRTUAL TABLE ft_post USING fts5(id, title, content);

    CREATE VIEW v_post_search AS
    SELECT
        p.id,
        json_object(
            'id', p.id,
            'title', p.title,
            'snippet', SUBSTRING(p.content, 1, 100),
            'rank', ft.rank
        ) AS data
    FROM tb_post p
    JOIN ft_post ft ON p.id = ft.id
    WHERE ft_post MATCH ?;
    ```
  </TabItem>

  <TabItem label="SQL Server">
    ```sql
    CREATE FULLTEXT INDEX ON dbo.tb_post(title, content)
    KEY INDEX PK_tb_post ON ft_catalog;

    CREATE VIEW dbo.v_post_search AS
    SELECT
        p.id,
        (
            SELECT
                p.id,
                p.title,
                SUBSTRING(p.content, 1, 100) AS snippet,
                ft.RANK AS rank
            FOR JSON PATH, WITHOUT_ARRAY_WRAPPER
        ) AS data
    FROM dbo.tb_post p
    JOIN FREETEXTTABLE(dbo.tb_post, (title, content), @searchterm) ft
        ON p.pk_post = ft.[KEY];
    ```
  </TabItem>
</Tabs>

None of this requires FraiseQL to "support" it. It's just SQL.

## Comparison With Other Approaches

| Aspect | ORM/Query Builder | GraphQL Auto-Generation | FraiseQL |
|--------|-------------------|------------------------|----------|
| SQL visibility | Hidden | Hidden | Written by you |
| Full SQL features | Subset | Subset | Everything your database offers |
| Performance debugging | Reverse-engineer | Reverse-engineer | `EXPLAIN ANALYZE` directly |
| AI generation | Complex (ORM DSL) | N/A | Simple (consistent pattern) |
| Learning curve | ORM-specific API | Framework-specific config | SQL + one pattern |
