Each walkthrough builds on the SQL-first model: `tb_` base tables → `v_` views with `jsonb_build_object` payloads → compile-time Python (or other-language) declarations that map to `fn_` mutation functions. The runtime is the same database you already know how to operate.

## Patterns covered across the walkthroughs

### Multi-tenancy

Row-Level Security on tenant-scoped tables; tenant context set from JWT claims via `current_setting('app.tenant_id')`. Covered in detail on the [SaaS Blog walkthrough](/examples/saas-blog) and on the dedicated [multi-tenancy guide](/building/multi-tenancy).

### Authentication and authorization

JWT-based auth with role enforcement inside SQL functions; `inject={"user_id": "jwt:sub"}` populates the caller identity at compile time so client-supplied identity is never trusted. Covered in the [authentication guide](/building/authentication).

### Soft deletes

`deleted_at TIMESTAMPTZ` columns plus views that filter `WHERE deleted_at IS NULL`. Restore is a one-statement update. Demonstrated in [SaaS Blog](/examples/saas-blog#soft-deletes--recovery).

### Real-time subscriptions

`pg_notify` from inside the mutation function; FraiseQL delivers the payload to GraphQL subscribers. No application-layer queue. Demonstrated in [Real-Time Collaboration](/examples/realtime-collaboration#real-time-subscriptions).

### Batch ingestion and rollups

A single SQL function ingests an array of events and updates rolling aggregates atomically — no per-event round-trip. Demonstrated in [Mobile Analytics Backend](/examples/mobile-analytics-backend#high-performance-event-ingestion).

## Cross-cutting code patterns

### Authentication and authorization

JWT claims drive both row-level filtering (`SET app.tenant_id = jwt.tenant_id`) and per-function role checks. The Python (or other-language) decorator is compile-time only — there is no runtime middleware to break.

```python
@fraiseql.query(requires_scope="read:posts")
def get_posts(user_id: ID) -> list[Post]:
    """Get posts visible to user."""
    pass
```

### Soft deletes

```python
@fraiseql.type
class Post:
    id: ID
    title: str
    deleted_at: datetime | None  # NULL = not deleted

@fraiseql.query
def get_posts() -> list[Post]:
    """Get non-deleted posts."""
    return fraiseql.config(sql_source="v_post")
```

### Cursor-based pagination

```python
@fraiseql.query
def get_posts(
    first: int = 10,
    after: str | None = None
) -> list[Post]:
    """Paginate through posts (relay connection returned automatically)."""
    return fraiseql.config(sql_source="v_post", relay=True)
```

### Error handling

Validation and error handling live in the SQL function, not in the Python decorator body. The Python layer is compile-time only:

```python
@fraiseql.mutation(sql_source="fn_create_post", operation="CREATE")
def create_post(title: str) -> Post:
    """Create post. fn_create_post validates title and returns an error status on failure."""
    pass
```

## REST and gRPC in the examples

All walkthroughs serve REST and gRPC alongside GraphQL on port 8080 — no extra configuration is needed. Each walkthrough notes which transport it exercises. To test REST or gRPC endpoints:

```bash
# REST — same resource names as GraphQL queries (default path: /rest/v1; configure via [rest] path)
curl http://localhost:8080/rest/v1/users

# gRPC — separate port (default 50052), requires grpcurl
grpcurl -plaintext localhost:50052 list
grpcurl -plaintext -d '{"limit": 10}' localhost:50052 fraiseql.BlogService/ListPosts
```

Schema annotations like `@fraiseql.query` (with `fraiseql.config(sql_source="v_post")` in the function body) automatically generate both the GraphQL resolver and the REST/gRPC endpoints. No separate REST or gRPC schema definition is needed.
