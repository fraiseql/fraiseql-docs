`fraiseql-wire` is a standalone Rust client library for streaming large JSON result sets from PostgreSQL with bounded memory. It connects directly to your database using the PostgreSQL wire protocol — a purpose-built alternative to general-purpose drivers when you need bulk reads without materializing the entire result set in memory.

<Aside type="note">
`fraiseql-wire` is a **separate crate** (`fraiseql-wire` on crates.io), not a configuration option in your FraiseQL server. You add it to your own Rust services that need to stream large datasets directly from the same PostgreSQL views that FraiseQL serves.
</Aside>

<Aside type="caution">
**Read-only by design.** `fraiseql-wire` does not support writes, mutations, transactions, or arbitrary SQL. Its only operation is `SELECT data FROM {view} [WHERE] [ORDER BY] [LIMIT] [OFFSET]`. For mutations, use the FraiseQL GraphQL endpoint.
</Aside>

## Scope

The FraiseQL wire protocol is a client library optimization for GraphQL transport. It is separate from the REST and gRPC transports — REST uses standard HTTP/JSON and gRPC uses standard Protobuf/HTTP2.

## When to Use Wire

| Scenario | Standard GraphQL | Wire |
|----------|-----------------|------|
| Interactive UI queries | Recommended | Overhead |
| < 10,000 rows | Recommended | Overkill |
| 10,000 – 100,000 rows | Acceptable | Recommended |
| > 100,000 rows | May OOM | Required |
| ETL pipelines | Awkward | Ideal |
| Bulk data exports | Memory issues | Ideal |
| Analytics workloads | — | Use Arrow instead |

## Installation

```bash
cargo add fraiseql-wire
```

Requires Rust 1.75+ and PostgreSQL 10+. No C dependencies — pure Rust via rustls (no OpenSSL required).

<Aside type="caution">
**Unix only.** `fraiseql-wire` supports Linux and macOS. Windows is not supported — the crate will fail to compile on non-Unix targets.
</Aside>

## Core API

### Connect

```rust
use fraiseql_wire::FraiseClient;

// Simplest: connection string
let client = FraiseClient::connect("postgres://user:pass@localhost/mydb").await?;

// With TLS
let client = FraiseClient::connect("postgres://user:pass@host/mydb?sslmode=require").await?;
```

### Query and stream

```rust
use futures::StreamExt;

let mut stream = client
    .query::<serde_json::Value>("orders")   // streams from v_orders view
    .execute()
    .await?;

while let Some(row) = stream.next().await {
    let json = row?;
    println!("{json}");
}
```

The generic type parameter affects only deserialization — it has no effect on the SQL sent to PostgreSQL.

### Type-safe deserialization

```rust
#[derive(serde::Deserialize)]
struct Order {
    id: String,
    total: f64,
    status: String,
}

let mut stream = client
    .query::<Order>("orders")
    .execute()
    .await?;

while let Some(result) = stream.next().await {
    let order: Order = result?;
    println!("Order {}: ${}", order.id, order.total);
}
```

## View Convention

`fraiseql-wire` reads from views named `v_{entity}` that expose a single column named `data` of type `json` or `jsonb`. This is the same naming convention FraiseQL uses:

```sql
-- Required view structure
CREATE VIEW v_orders AS
SELECT jsonb_build_object(
    'id',     o.id,
    'total',  o.total,
    'status', o.status
) AS data
FROM tb_order o;
```

`tv_{entity}` (time-series views) are intentionally not supported.

## Filtering

Wire supports hybrid filtering — SQL predicates reduce wire traffic, Rust predicates refine locally:

```rust
let stream = client
    .query::<serde_json::Value>("orders")
    // Evaluated in PostgreSQL — reduces rows returned
    .where_sql("data->>'status' = 'shipped'")
    // Evaluated in Rust — applied after PostgreSQL delivers rows
    .where_rust(|json| {
        json["total"].as_f64().unwrap_or(0.0) > 500.0
    })
    .execute()
    .await?;
```

### Type-safe operators

For complex predicates, use the `WhereOperator` enum instead of raw SQL strings:

```rust
use fraiseql_wire::operators::{WhereOperator, Field, Value};

let filter = WhereOperator::Gt(
    Field::new("data->>'total'"),
    Value::Float(500.0),
);

let stream = client
    .query::<serde_json::Value>("orders")
    .where_operator(filter)
    .execute()
    .await?;
```

25+ operators are available, including array operations, string matching (ILIKE, LIKE), null checks, pgvector distances (L2, cosine, inner product), full-text search, and IP network operations.

## Ordering and Pagination

```rust
let stream = client
    .query::<serde_json::Value>("orders")
    .order_by("data->>'created_at' DESC")
    .limit(1000)
    .offset(500)
    .execute()
    .await?;
```

With COLLATE for locale-aware string sorting:

```rust
.order_by("data->>'name' ASC COLLATE \"C\"")
```

### Reduce payload size with projections

```rust
// Only request the fields you need — reduces wire traffic
.select_projection(
    "jsonb_build_object('id', data->>'id', 'total', data->>'total')"
)
```

## Memory Management

`fraiseql-wire` maintains `O(chunk_size)` memory regardless of result size. A 1-million-row query uses the same memory as a 100-row query.

```rust
let stream = client
    .query::<serde_json::Value>("large_table")
    .chunk_size(512)              // Items buffered at once (default: 256)
    .max_memory(500_000_000)      // 500 MB hard limit — returns error if exceeded
    .adaptive_chunking(true)      // Auto-optimize based on channel pressure (default: on)
    .adaptive_min_size(16)        // Never shrink below 16 items
    .adaptive_max_size(1024)      // Never grow above 1024 items
    .execute()
    .await?;
```

### Benchmarked memory usage

| Rows | tokio-postgres | fraiseql-wire |
|------|---------------|---------------|
| 10,000 | 2.6 MB | **1.3 KB** |
| 100,000 | 26 MB | **1.3 KB** |
| 1,000,000 | 260 MB | **1.3 KB** |

Memory stays constant at approximately `chunk_size × avg_row_size`.

## Stream Control

For advanced flow control — for example, to throttle consumption or coordinate with a downstream system:

```rust
let mut stream = client.query::<T>("orders").execute().await?;

// Process some rows
while let Some(item) = stream.next().await {
    process(item?);
    break;
}

// Pause production — PostgreSQL stops sending rows
stream.pause().await?;

// ... do other work ...

// Resume — continues from exactly where it stopped
stream.resume().await?;

// Inspect current stream state
let stats = stream.stats();
println!("Rows yielded: {}", stats.rows_yielded);
```

## Configuration

### Connection string parameters

```
postgres://[user[:password]@][host][:port][/database][?param=value...]
```

Supported parameters:

| Parameter | Example | Description |
|-----------|---------|-------------|
| `sslmode` | `require` | `disable`, `require`, `verify-ca`, `verify-full` |
| `sslrootcert` | `/path/to/ca.pem` | Custom CA certificate |
| `sslcert` | `/path/to/cert.pem` | Client certificate (mTLS) |
| `sslkey` | `/path/to/key.pem` | Client key (mTLS) |
| `application_name` | `my-service` | Visible in `pg_stat_activity` |
| `statement_timeout` | `30000ms` | Per-query timeout |
| `tcp_keepalives_idle` | `300s` | Keep-alive interval |

Unix socket connection: `postgres:///mydb`

### ConnectionConfig builder

For more control than connection strings allow:

```rust
use fraiseql_wire::ConnectionConfig;
use std::time::Duration;

let config = ConnectionConfig::builder("localhost", "mydb")
    .password("secret")
    .connect_timeout(Duration::from_secs(10))
    .statement_timeout(Duration::from_secs(30))
    .keepalive_idle(Duration::from_secs(300))
    .application_name("my-service")
    .build();

let client = FraiseClient::connect_with_config("postgres://host/mydb", config).await?;
```

### TLS configuration

```rust
use fraiseql_wire::TlsConfig;

// Custom CA (server certificate verification)
let tls = TlsConfig::builder()
    .verify_hostname(true)
    .ca_cert_path("/path/to/ca.pem")?
    .build()?;

// Mutual TLS
let tls = TlsConfig::builder()
    .ca_cert_path("/path/to/ca.pem")?
    .client_cert_path("/path/to/cert.pem")?
    .client_key_path("/path/to/key.pem")?
    .build()?;

let client = FraiseClient::connect_tls("postgres://host/db", tls).await?;
```

Authentication is SCRAM-SHA-256 with channel binding (prevents MITM attacks). Implemented in pure Rust via rustls — no OpenSSL dependency.

## Error Handling

```rust
use fraiseql_wire::Error;

match client.query::<T>("orders").execute().await {
    Err(Error::Connection(msg)) => eprintln!("Connection failed: {msg}"),
    Err(Error::Authentication(msg)) => eprintln!("Auth failed: {msg}"),
    Err(Error::Sql(msg)) => eprintln!("SQL error: {msg}"),
    Err(Error::MemoryLimitExceeded { limit, estimated_memory }) => {
        eprintln!("Memory limit {limit} exceeded (estimated {estimated_memory})")
    }
    Err(Error::Cancelled) => { /* stream was dropped */ }
    Ok(stream) => { /* use stream */ }
}
```

## Metrics

`fraiseql-wire` emits metrics via the `metrics` crate (framework-agnostic). Register a recorder (Prometheus, StatsD, etc.) before connecting:

| Metric | Type | Description |
|--------|------|-------------|
| `fraiseql_rows_processed_total` | Counter | Rows delivered to caller per entity |
| `fraiseql_rows_filtered_total` | Counter | Rows dropped by Rust predicate |
| `fraiseql_query_total_duration_ms` | Histogram | Full query duration |
| `fraiseql_query_startup_duration_ms` | Histogram | Time to first row |
| `fraiseql_chunk_processing_duration_ms` | Histogram | Per-chunk latency |
| `fraiseql_memory_limit_exceeded_total` | Counter | Memory limit exceeded events per entity |
| `fraiseql_chunk_size_bytes` | Gauge | Active chunk size in bytes (adaptive) |
| `fraiseql_stream_buffered_items` | Gauge | Items waiting in buffer |
| `fraiseql_query_error_total` | Counter | Errors by entity and category |

## Comparison with Arrow

| Feature | Wire | Arrow |
|---------|------|-------|
| Format | JSON (row-oriented) | Columnar binary |
| Memory | Bounded (O(chunk_size)) | Bounded |
| Speed | 100K–500K rows/sec | Higher throughput |
| Compatibility | Any Rust async code | Arrow Flight SDK |
| Use case | ETL, bulk export, services | Analytics, ML, data science |
| Client languages | Rust | Python, TypeScript, Go, Rust, R, Julia |

<Aside type="tip">
Use **Wire** when your downstream consumer is a Rust service and you need JSON output. Use **Arrow** when your consumer is a data science or analytics tool that benefits from columnar layout.
</Aside>

## Troubleshooting

### Connection refused

```
Error::Connection("Connection refused")
```

1. Verify PostgreSQL is running and accessible
2. Check host, port, and database name in connection string
3. Confirm the user exists and has `SELECT` privileges on the view

### Authentication failure

```
Error::Authentication("SCRAM authentication failed")
```

1. Verify username and password
2. Check `pg_hba.conf` allows SCRAM-SHA-256 for this user and host
3. PostgreSQL 10+ required for SCRAM-SHA-256

### `data` column not found

```
Error::InvalidSchema("Expected column 'data', got: ...")
```

The view must expose exactly one column named `data` of type `json` or `jsonb`. See [View Convention](#view-convention) above.

### Memory limit exceeded

```
Error::MemoryLimitExceeded { limit, estimated_memory }
```

Increase `max_memory()` or reduce `chunk_size()` to lower per-chunk footprint:

```rust
.chunk_size(64)
.max_memory(1_000_000_000)  // 1 GB
```

### Slow time-to-first-row

1. Verify indexes exist on the `WHERE` columns in the underlying table
2. Run `EXPLAIN ANALYZE` on the view query
3. Use `.select_projection(...)` to reduce row payload size

### Stream hangs mid-result

1. Check `statement_timeout` — set it higher than your longest expected export:
   ```
   postgres://host/db?statement_timeout=600000ms
   ```
2. Check for intermediate proxies (nginx, ALB) closing long-lived connections
