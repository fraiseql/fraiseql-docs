The FraiseQL Wire protocol provides memory-efficient streaming for large query results, using PostgreSQL's COPY protocol for minimal overhead.

<Aside type="caution">
The Wire protocol uses **PostgreSQL's binary COPY protocol** for data transfer and is only supported with a PostgreSQL database backend. It is not available with other database backends.
</Aside>

## Overview

Traditional query execution loads entire result sets into memory. The Wire protocol streams data in chunks, maintaining bounded memory regardless of result size.

Wire protocol is ideal for:
- Exporting large datasets
- ETL pipelines
- Reports with millions of rows
- Memory-constrained environments

## When to Use Wire

| Scenario | Standard | Wire |
|----------|----------|------|
| < 10,000 rows | Recommended | Overkill |
| 10,000 - 100,000 rows | Acceptable | Recommended |
| > 100,000 rows | May OOM | Required |
| Real-time queries | Recommended | Overhead |
| Batch exports | Memory issues | Ideal |

## Configuration

### Enable Wire Backend

```toml
[server]
backend = "wire"

[wire]
# Connection settings
connection_timeout_ms = 5000
query_timeout_ms = 300000  # 5 minutes for large queries

# Chunking
adaptive_chunking = true
initial_chunk_size = 1000
max_chunk_size = 10000
target_memory_mb = 100
```

### Stream Timeout Configuration

For large exports, tune timeouts to prevent premature disconnection:

```toml
[wire]
# Time to establish a connection to PostgreSQL
connection_timeout_ms = 10000

# Maximum time a single query may run (including streaming)
# Set this based on your largest expected export
query_timeout_ms = 600000   # 10 minutes for bulk exports

# How long to wait for the next chunk before aborting the stream
# Increase if PostgreSQL is slow to produce rows (e.g., complex aggregations)
chunk_idle_timeout_ms = 30000
```

<Aside type="tip">
If clients report truncated responses on large exports, increase `query_timeout_ms` first. The default 5-minute timeout is conservative for multi-million-row exports.
</Aside>

### Feature Flag

Wire is an optional feature:

```bash
# Build with wire support
cargo build --features wire-backend
```

## How It Works

### PostgreSQL COPY Protocol

Wire uses PostgreSQL's binary COPY protocol for efficient data transfer:

```
PostgreSQL Server ──(COPY TO STDOUT binary row stream)──> FraiseQL Wire ──> Chunked JSON
```

### Adaptive Chunking

Wire dynamically adjusts chunk sizes based on row size:

```rust
// Small rows (< 1KB): larger chunks
chunk_size = 10_000

// Large rows (10KB+): smaller chunks
chunk_size = 100

// Memory target maintained regardless of row size
```

## Performance Benchmarks

The Wire protocol's throughput advantage over standard JSON comes from avoiding full result materialization. Representative numbers on commodity hardware (8-core, 16 GB RAM, PostgreSQL on the same host):

| Workload | Standard GraphQL | Wire Protocol |
|----------|-----------------|---------------|
| 10K rows, 10 columns, ~500 B/row | ~30 MB/s, ~350 MB peak RAM | ~120 MB/s, ~12 MB peak RAM |
| 100K rows, 10 columns, ~500 B/row | ~25 MB/s, ~3.2 GB peak RAM | ~200 MB/s, ~12 MB peak RAM |
| 1M rows, 10 columns, ~500 B/row | OOM (> 32 GB) | ~220 MB/s, ~12 MB peak RAM |
| 10K rows, wide (50 columns, ~5 KB/row) | ~8 MB/s, ~3.5 GB peak RAM | ~90 MB/s, ~18 MB peak RAM |

Throughput in rows/sec at 500 B/row average:

| Configuration | Rows/sec |
|---------------|----------|
| `initial_chunk_size = 1000` | ~180,000 rows/sec |
| `max_chunk_size = 10000` | ~440,000 rows/sec |
| `max_chunk_size = 10000` + `compression = "zstd"` | ~380,000 rows/sec (30% smaller payload) |

## Client Usage

### Streaming Response

Wire returns newline-delimited JSON:

```bash
curl -N http://localhost:8080/graphql \
    -H "Content-Type: application/json" \
    -H "Accept: application/x-ndjson" \
    -d '{"query": "{ allOrders { id total } }"}'
```

Response (streamed):

```
{"data":{"allOrders":[{"id":"1","total":99.99}
{"id":"2","total":149.99}
{"id":"3","total":249.99}
...
]}}
```

### Streaming Client Examples

<Tabs syncKey="language">
<TabItem label="JavaScript">
```javascript
async function* streamQuery(query) {
    const response = await fetch('/graphql', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'Accept': 'application/x-ndjson'
        },
        body: JSON.stringify({ query })
    });

    const reader = response.body.getReader();
    const decoder = new TextDecoder();
    let buffer = '';

    while (true) {
        const { done, value } = await reader.read();
        if (done) break;

        buffer += decoder.decode(value, { stream: true });
        const lines = buffer.split('\n');
        buffer = lines.pop() || '';

        for (const line of lines) {
            if (line.trim()) {
                yield JSON.parse(line);
            }
        }
    }
}

// Usage
for await (const row of streamQuery('{ allOrders { id total } }')) {
    console.log('Row:', row);
}
```

import httpx
import json

def stream_query(query: str):
    with httpx.stream(
        "POST",
        "http://localhost:8080/graphql",
        json={"query": query},
        headers={"Accept": "application/x-ndjson"}
    ) as response:

            if line.strip():
                yield json.loads(line)

# Usage
for row in stream_query("{ allOrders { id total } }"):

<TabItem label="Go">
```go
package main

import (
    "bufio"
    "bytes"
    "encoding/json"
    "fmt"
    "net/http"
)

type Row map[string]any

func streamQuery(query string) (<-chan Row, error) {
    payload, _ := json.Marshal(map[string]string{"query": query})

    req, err := http.NewRequest(http.MethodPost, "http://localhost:8080/graphql",
        bytes.NewReader(payload))
    if err != nil {
        return nil, err
    }
    req.Header.Set("Content-Type", "application/json")
    req.Header.Set("Accept", "application/x-ndjson")

    resp, err := http.DefaultClient.Do(req)
    if err != nil {
        return nil, err
    }

    ch := make(chan Row, 100)
    go func() {
        defer close(ch)
        defer resp.Body.Close()

        scanner := bufio.NewScanner(resp.Body)
        scanner.Buffer(make([]byte, 64*1024), 1024*1024) // 1 MB max line
        for scanner.Scan() {
            line := scanner.Bytes()
            if len(bytes.TrimSpace(line)) == 0 {
                continue
            }
            var row Row
            if err := json.Unmarshal(line, &row); err == nil {
                ch <- row
            }
        }
    }()

    return ch, nil
}

func main() {
    rows, err := streamQuery("{ allOrders { id total } }")
    if err != nil {
        panic(err)
    }

    count := 0
    for row := range rows {
        count++
        if count <= 3 {
            fmt.Printf("Row %d: %v\n", count, row)
            // Row 1: map[id:order-001 total:99.99]
            // Row 2: map[id:order-002 total:149.99]
            // Row 3: map[id:order-003 total:249.99]
        }
    }
    fmt.Printf("Total rows received: %d\n", count)
}
```

## Memory Management

### Bounded Memory

Wire maintains bounded memory regardless of result size:

```toml
[wire]
target_memory_mb = 100  # Stay under 100MB
```

Memory usage stays constant:

| Rows | Standard Memory | Wire Memory |
|------|-----------------|-------------|
| 10,000 | 50 MB | ~10 MB |
| 100,000 | 500 MB | ~10 MB |
| 1,000,000 | 5 GB (OOM!) | ~10 MB |

### Memory Estimation

Wire estimates row sizes to prevent OOM:

```rust
// Estimates before fetching
estimated_row_size = avg(sample_rows)
safe_chunk_size = target_memory / estimated_row_size
```

## Authentication

Wire supports SCRAM-SHA-256 authentication:

```toml
[wire.auth]
method = "scram-sha-256"
username = "${DB_USER}"
password = "${DB_PASSWORD}"
```

## TLS Support

```toml
[wire.tls]
mode = "require"  # disable, prefer, require
ca_cert = "/path/to/ca.crt"
```

## Comparison with Arrow

| Feature | Wire | Arrow |
|---------|------|-------|
| Format | JSON (NDJSON) | Columnar binary |
| Memory | Bounded | Bounded |
| Speed | Fast | Faster |
| Compatibility | Universal (HTTP + JSON) | Arrow Flight SDK required |
| Use case | Web clients, ETL | Analytics, ML |
| Client languages | Any | Python, TypeScript, Go, Rust, R, Julia |

<Aside type="tip">
Use **Wire** for JSON-based clients, simple streaming exports, and maximum compatibility. Use **Arrow** for analytics workloads, data science pipelines, and maximum performance.
</Aside>

## Best Practices

### Set Appropriate Timeouts

```toml
[wire]
query_timeout_ms = 300000  # 5 min for exports
connection_timeout_ms = 10000
chunk_idle_timeout_ms = 30000
```

### Monitor Memory

```txt
# Alert if memory exceeds target
fraiseql_wire_memory_usage > 150_000_000
```

### Use for Exports Only

Don't use Wire for:
- Real-time queries (use standard)
- Small result sets (overhead not worth it)
- Interactive UIs (latency-sensitive)

## Troubleshooting

### Connection Timeouts

```toml
[wire]
connection_timeout_ms = 30000  # Increase timeout
```

### Slow Streaming

1. Check network latency to database
2. Verify indexes on queried tables
3. Increase chunk size for small rows

### Memory Spikes

1. Reduce `target_memory_mb`
2. Enable `adaptive_chunking`
3. Check for unexpectedly large rows

### Stream Truncated Mid-Response

1. Increase `query_timeout_ms` — the most common cause for large exports
2. Check `chunk_idle_timeout_ms` if PostgreSQL has slow row production
3. Verify no intermediate proxy (nginx, ALB) is terminating the long-running HTTP connection

## Troubleshooting

### Stream Truncated

If exports cut off mid-stream:

1. **Increase query timeout**:
   ```toml
   [wire]
   query_timeout_ms = 600000  # 10 minutes
   ```

2. **Check proxy timeouts** (nginx, ALB, etc.):
   ```nginx
   proxy_read_timeout 600s;  # Match query_timeout_ms
   proxy_send_timeout 600s;
   ```

3. **Verify PostgreSQL isn't timing out**:
   ```sql
   SHOW statement_timeout;
   -- Set to 0 (unlimited) or higher than FraiseQL timeout
   SET statement_timeout = '10min';
   ```

### Slow Streaming

If throughput is lower than expected:

1. **Check PostgreSQL performance**:
   ```sql
   EXPLAIN ANALYZE SELECT id, total, status FROM tb_order;
   ```

2. **Increase chunk size for small rows**:
   ```toml
   [wire]
   initial_chunk_size = 10000
   max_chunk_size = 50000
   ```

3. **Verify network bandwidth**:
   ```bash
   iperf3 -c localhost -p 8080
   ```

### Connection Errors

```
ERROR: Wire protocol connection failed
```

1. Verify PostgreSQL is accessible
2. Check credentials in `DATABASE_URL`
3. Ensure `pgcrypto` extension is installed:
   ```sql
   CREATE EXTENSION IF NOT EXISTS pgcrypto;
   ```
