FraiseQL ships an observer runtime as the `fraiseql-observers` crate. The `fraiseql-server` binary auto-instantiates the runtime when the `observers` Cargo feature is built in, mounts the management HTTP API at `/api/observers`, and polls the `tb_entity_change_log` table for INSERT / UPDATE / DELETE events. Each event is matched against rows in the `tb_observer` table and dispatched through the registered actions (webhook, Slack, email stub, future variants) with retry and an in-process dead-letter queue. The runtime is auto-wired, but **the v2.3.2 surface ships with two critical security regressions and four operational regressions documented below — read [Security caveats](#security-caveats) before exposing the management API to any caller**, then see [Known issues](#known-issues) for the operational gaps.

## Security caveats

Eight framework issues are open against the v2.3.2 observer surface; four are security-class and the remaining four are operational regressions covered under [Known issues](#known-issues). **Do not expose the observer management API or the runtime routes to the public internet without the mitigations below.** The combination of FW-20 and FW-21 below is a one-step path to IAM-credential exfiltration on a cloud-hosted deployment.

### FW-20 [#347](https://github.com/fraiseql/fraiseql/issues/347) + FW-21 [#348](https://github.com/fraiseql/fraiseql/issues/348) — anonymous admin API + SSRF-bypass env var (critical, combined)

FW-21 [#348](https://github.com/fraiseql/fraiseql/issues/348) — every observer management endpoint accepts anonymous requests at v2.3.2. The router takes no `auth::Claims` extractor, performs no `state.rbac.can_*` check, and the binary mounts the routes unconditionally when the `observers` feature is built in. `POST /api/observers` creates an observer, `POST /runtime/reload` reloads the runtime, `POST /api/observers/dlq/{id}/retry` re-fires a failed event, `DELETE /api/observers/{id}` removes one, all without authentication.

{/* source: crates/fraiseql-server/src/server/routing/observers.rs:L18-L66 — routes mounted unconditionally when observers feature is built in */}
{/* source: crates/fraiseql-server/src/observers/routes.rs — handler signatures take no auth extractor */}

FW-20 [#347](https://github.com/fraiseql/fraiseql/issues/347) — the SSRF allowlist in `validate_outbound_url` (which rejects loopback, RFC 1918, link-local, CGNAT, and ULA destinations on every webhook URL) has a global kill-switch: `FRAISEQL_OBSERVERS_ALLOW_INSECURE=true` disables every guard in nine call sites and emits a one-shot WARN at startup. There is no production-mode safeguard, no per-environment refusal, no `RUST_LOG` audit on subsequent uses.

{/* source: crates/fraiseql-observers/src/actions.rs:L33-L80 — validate_outbound_url + FRAISEQL_OBSERVERS_ALLOW_INSECURE kill-switch */}
{/* source: crates/fraiseql-observers/src/actions.rs:L25-L28 — INSECURE_WARN_ONCE: Once */}

**Combined exploitation path** (no authentication required at v2.3.2):

1. Attacker `POST /api/observers` to register a new observer with `action = "webhook"` and `url = "http://169.254.169.254/latest/meta-data/iam/security-credentials/"` (the AWS EC2 instance metadata endpoint; analogous URLs exist on GCP, Azure, and DigitalOcean).
2. Operator (or attacker via a triggering mutation) causes the matching entity event to fire.
3. The webhook action fetches the IAM credentials and POSTs them at the destination of the attacker's choice via subsequent retries or a second observer.

The SSRF allowlist is the only guard preventing step 1 from succeeding. With `FRAISEQL_OBSERVERS_ALLOW_INSECURE=true` set anywhere in the environment, that guard is silently removed.

**Mitigations (apply all three at v2.3.2):**

- Gate `/api/observers/*` and `/runtime/*` behind an authenticating reverse proxy (mTLS, OIDC, or a known-key header check) so the framework's missing auth gate is closed at the network layer. Treat the management API as a privileged plane.
- Never set `FRAISEQL_OBSERVERS_ALLOW_INSECURE=true` in any environment with network reachability to RFC 1918 / link-local addresses, including local dev where a stray `export` can leak into a deployment.
- Restrict the observer management API to a VPN-only network or a private subnet at v2.3.2. Re-evaluate when FW-20 + FW-21 land upstream.

### FW-18 [#345](https://github.com/fraiseql/fraiseql/issues/345) — webhook payloads are not signed (no HMAC)

`WebhookAction::execute` POSTs the rendered body with no signature. The crate has zero hits for `hmac`, `signature`, `X-Webhook-Signature`, or `X-FraiseQL-Signature`, and pulls no `hmac` dependency (the `sha2` import is for cache-key hashing only). Webhook receivers cannot cryptographically verify the request originated from FraiseQL.

{/* source: crates/fraiseql-observers/src/actions.rs:L240-L320 — WebhookAction::execute sends unsigned POST */}
{/* source: crates/fraiseql-observers/Cargo.toml — no hmac dep at frozen SHA */}

**Mitigation:** terminate webhook subscriptions at an authenticating gateway that injects a bearer token or re-signs the body with a key the destination knows. Do not rely on origin-IP allowlisting alone — FW-20 + FW-21 above plus a public network give an attacker the ability to send any payload from any address the framework reaches.

### FW-19 [#346](https://github.com/fraiseql/fraiseql/issues/346) — webhook URL, headers, and body logged at INFO (PII / bearer-token leak)

Four `info!` log lines on every webhook dispatch expose the full webhook URL, the entire `HashMap<String, String>` headers map (verbatim `Debug` format — including `Authorization: Bearer …` and `X-API-Key: …` if the operator put them there), the raw body template, and the rendered JSON body. There is no redact / mask helper, no env-var toggle, no feature-flag gate. Centralised log aggregators ingest every event payload at the default INFO level.

{/* source: crates/fraiseql-observers/src/actions.rs:L254-L290 — four info! lines on every webhook dispatch */}

**Mitigation:** set `RUST_LOG=fraiseql_server=warn,fraiseql_observers=warn` in production so the INFO-level lines do not write. Do not interpolate PII into webhook URLs or header values, even if the destination requires a bearer token — emit the bearer at the gateway layer instead so the framework never sees it.

### FW-22 [#349](https://github.com/fraiseql/fraiseql/issues/349) — `EmailAction` is a stub that reports success

The `Email` action variant deserialises from `ActionConfig` and dispatches through `EmailAction::execute`, but the executor returns `Ok(())` without sending an email. No SMTP connection, no MX lookup, no message construction. Observability counters increment as if delivery succeeded; `fraiseql_observer_action_executed_total` shows 100 % delivery, `fraiseql_observer_action_errors_total` stays at zero, and the DLQ stays empty. Silent failure with green metrics.

{/* source: crates/fraiseql-observers/src/actions.rs — EmailAction::execute returns Ok without sending */}

**Mitigation:** until fixed, route email notifications through a webhook handler to a real email gateway (Resend, SendGrid, Postmark, Mailgun, or an internal SMTP relay). Do not configure `action = "email"` in any production observer at v2.3.2.

## Mental model

An observer is a row in the `tb_observer` PostgreSQL table that maps an `(entity_type, event_type)` pair to one or more actions. When a tracked entity row is INSERTed, UPDATEed, or DELETEed, the runtime publishes a CDC event to `tb_entity_change_log`, the matcher resolves the observers that subscribe to that pair, and each observer's actions execute through the executor with retry / DLQ behaviour configured per action.

```
mutation → tb_entity_change_log → ChangeLogListener → matcher → executor → action(s)
                                                                                 ↓ on failure
                                                                            retry → DLQ
```

{/* source: crates/fraiseql-observers/src/listener/change_log.rs:L1 — PG poll loop entrypoint */}
{/* source: crates/fraiseql-observers/src/executor/dispatch.rs:L1-L60 — dispatch path */}

The runtime is composed of seven primary subsystems, all visible in the crate's `lib.rs` module rustdoc: `listener` (CDC poll + lease), `matcher` (Elo condition evaluator + `entity_type_index`), `executor` (dispatch + retry + DLQ), `actions` (webhook / Slack / email / future variants), `job_queue` (async worker pool), `metrics` (Prometheus registry), and `transport` (Postgres LISTEN/NOTIFY, NATS, or in-memory bridge). The off-the-shelf binary wires the Postgres transport by default; the others require Cargo features and library-API composition (see [Backends and transports](#backends-and-transports) below — FW-23 silently disables the TOML transport selector).

## How observers are composed today

## Backends and transports

`TransportKind` (the library-API enum) has three variants: `Postgres` (default), `Nats`, and `InMemory`. Redis is **not** an event transport — it backs the observer cache, dedup, job queue, and lease subsystems. MySQL appears only as a one-way bridge (MySQL CDC → NATS), not a standalone event transport.

## Action types

`ActionConfig` is a `#[non_exhaustive]` tagged union (tag = `"type"`, snake_case). Four variants ship: `Webhook`, `Slack`, `Email` (stub — FW-22), and an internal NATS-publish path. Three additional variants exist as stubs: `Sms`, `Push`, `Search`, `Cache`. The `non_exhaustive` attribute is a forward-compatibility guarantee for the framework, **not** a user-pluggable hook — operators cannot register a custom action type from outside the crate.

### Webhook

```json
{
  "type": "webhook",
  "url_env": "MY_WEBHOOK_URL",
  "headers": {"X-Source": "fraiseql"},
  "body_template": "{\"id\": \"{{id}}\", \"total\": {{total}}}"
}
```

The webhook action POSTs the rendered body to the configured URL. A `Mustache`-style template renders `{{field_name}}` substitutions against the event payload. The default HTTP timeout is 30 seconds (`DEFAULT_WEBHOOK_TIMEOUT_SECS`). Outbound URLs are validated against an SSRF allowlist (`validate_outbound_url`) on every dispatch — see [Security caveats](#security-caveats) for the FW-20 kill-switch that disables the allowlist globally.

{/* source: crates/fraiseql-observers/src/actions.rs:L20-L23 — DEFAULT_WEBHOOK_TIMEOUT_SECS = 30 */}

The retry policy is configured per observer (not per action) via the `retry` field on the observer row. Supported strategies: `Exponential` (default), `Linear`, `Fixed`. Each strategy applies ±25 % jitter and saturating arithmetic. Non-2xx responses are classified: 4xx (except 429) → permanent → DLQ, 5xx and 429 → transient → retry.

## Lifecycle: subscription → dispatch → retry → DLQ

1. **Subscribe.** `POST /api/observers` writes a row to `tb_observer`. The runtime's `auto_reload` loop picks it up on the next `reload_interval_secs` boundary (default 60 seconds), or call `POST /runtime/reload` to force an immediate reload.
2. **Mutate.** Any mutation that touches a tracked entity writes a row to `tb_entity_change_log` via the entity-changelog trigger.
3. **Listen.** `ChangeLogListener` polls `tb_entity_change_log` every `poll_interval_ms` (default 100 ms), draining up to `batch_size` (default 100) rows per iteration.
4. **Match.** The matcher resolves the `(entity_type, event_type)` pair against the `entity_type_index` ArcSwap snapshot, evaluating each candidate observer's `condition` against the event payload through the Elo DSL evaluator.
5. **Dispatch.** Each matched observer's actions are queued on the in-process channel (capacity `channel_capacity`, default 1000). The `OverflowPolicy::Drop` default (library-side) drops events when the channel is full and increments `fraiseql_observer_backlog_size`.
6. **Execute.** Worker tasks pull from the channel and call each `ActionConfig`'s executor. Each action runs to completion or to its configured retry budget.
7. **Retry.** Failed dispatches re-queue with the `RetryConfig`'s backoff strategy until `max_attempts` is reached (default 3).
8. **DLQ.** Events that exhaust their retries land in the in-process `InMemoryDlq`, exposed at `GET /api/observers/dlq`. **The DLQ is unbounded** (FW-16 [#343](https://github.com/fraiseql/fraiseql/issues/343)) and lost on process restart.

## `entity_type_index` atomicity (F056)

The matcher's `(entity_type, event_type) → [observer_id]` lookup table is held in an `Arc<ArcSwap<HashMap<...>>>`. Reload republishes the whole map via a single atomic `store(Arc::new(new_map))`; readers do `.load().get(...)` on a snapshot Arc. A concurrent CDC dispatch observes either the pre-reload generation or the post-reload generation atomically — never a partial or empty index. This is the F056 guarantee.

{/* source: crates/fraiseql-server/src/observers/runtime.rs:L18 — use arc_swap::ArcSwap */}
{/* source: crates/fraiseql-server/src/observers/runtime.rs:L139 — entity_type_index field */}
{/* source: crates/fraiseql-server/src/observers/runtime.rs:L164 — Arc::new(ArcSwap::from_pointee(HashMap::new())) */}
{/* source: crates/fraiseql-server/src/observers/runtime.rs:L300 — store(Arc::new(...)) on initial load */}
{/* source: crates/fraiseql-server/src/observers/runtime.rs:L673 — store(Arc::new(...)) on reload */}
{/* source: CHANGELOG.md:L347-L358 — F006/F007/F008/F013/F048/F056/F057 batch */}

:::note[Brief generation skew on reload — page-level guidance, not a bug]
The sibling `matcher: Arc<RwLock<Option<EventMatcher>>>` and `executor: Arc<RwLock<Option<Arc<ObserverExecutor>>>>` are written sequentially during reload (matcher first, then executor, then the `entity_type_index` ArcSwap). An in-flight dispatch can observe (new matcher, old executor) for a brief window — but each component is internally consistent and the dispatch path does not cross-validate them, so reload-during-dispatch is at worst a one-event "generation skew" and never a panic or a lost event. The F056 atomicity guarantee covers the index lookup; matcher/executor pairs are not lock-free but are individually consistent.
:::

## Worker panic propagation (F014)

The async worker pool catches panics at the `JoinHandle` layer. When a worker task panics, `handle_join_outcome` matches `JoinError::is_panic()`, logs at `error!` with `worker=` / `error=` fields, increments `fraiseql_observer_job_failed_total{error="panic"}`, and the executor loop continues. The runtime does not deadlock; the next event in the queue dispatches normally.

## Cross-instance HA via `observers-enterprise`

The `observers-enterprise` server feature pulls `fraiseql-observers/{enterprise, nats}`. The `enterprise` bundle on the crate side is `["checkpoint", "dedup", "caching", "queue", "search", "metrics"]`; `caching` transitively pulls `redis`, which makes the `redis-lease` backend available. With the `redis-lease` backend wired in a host binary, multiple FraiseQL replicas coordinate so only one instance drives the listener at a time — the lease holder polls CDC, dispatches actions, and writes checkpoints; on lease loss (TTL expiry, network partition, or graceful release), another replica takes over.

{/* source: crates/fraiseql-server/Cargo.toml:L175 — observers-enterprise = ["observers", "fraiseql-observers/enterprise", "fraiseql-observers/nats"] */}
{/* source: crates/fraiseql-observers/Cargo.toml:L57 — enterprise = ["checkpoint", "dedup", "caching", "queue", "search", "metrics"] */}
{/* source: crates/fraiseql-observers/src/listener/lease.rs:L1-L18 — in-process, postgres advisory lock, redis SET NX EX */}

Without `redis-lease` or the Postgres advisory-lock backend wired by the host binary, the default lease is in-process — every replica's runtime polls and dispatches independently, and any event is processed once per replica. At-least-once delivery still holds within a single replica; cross-replica deduplication requires the `dedup` feature plus a Redis instance.

| `app.nest("/api/observers", observer_routes(...))` | `GET\|POST\|PUT\|DELETE /api/observers[...]` | Observer CRUD against `tb_observer` |
| `app.nest("/api/observers", observer_changelog_routes(...))` | `GET /api/observers/changelog`, `GET\|PUT /api/observers/checkpoint/{id}` | Changelog read + checkpoint persistence |
| `app.nest("/api/observers", observer_dlq_routes(...))` | `GET /api/observers/dlq`, `POST /api/observers/dlq/{id}/retry`, `POST /api/observers/dlq/retry-all`, `DELETE /api/observers/dlq/{id}` | DLQ inspection + retry |
| `app.nest("/api/observers", observer_dlq_routes(...))` (same router) | `GET /api/observers/delivery/health` | DLQ delivery health summary |

## Worked example

The companion script `scripts/docs-test/pages/observers.docs-test.sh` documents the intended host-binary flow:

1. Bring up FraiseQL + PostgreSQL with the observers overlay; apply `migrations/06_create_observer_management.sql` so `tb_observer` and the DLQ schema exist.
2. Confirm `/health` flips from `degraded` to `healthy`; `GET /runtime/health` returns 200 with `running=true`.
3. `POST /api/observers` with a webhook observer pointing at a netcat sink in the test container.
4. Trigger the source event via a tracked mutation.
5. Verify the webhook payload reaches the sink.
6. Kill the sink; verify retries; verify DLQ accumulates the failed event.

At v2.3.2 the script does **not** drive the documented end-to-end happy path because FW-23 silently keeps the Postgres listener and FW-15 splits the TOML between CLI validation and runtime knobs. When the framework fixes ship and the binary wires the dispatch flow, the script's "still broken" assertions flip and the test fails loudly — the regression signal Phase 09 needs.

## Migration from the pre-v2.3 page

:::caution[Previous version: the Python `@observer` decorator and `actions=[slack(), email()]` kwargs do not exist]
The pre-v2.3 revision of this page documented a Python `@fraiseql.observer` decorator with `entity=`, `event=`, `condition=`, and `actions=[slack(...), email(...), webhook(...)]` keyword arguments, plus a `RetryConfig` Python class. **None of those symbols exist in the `fraiseql` Python package at v2.3.2.** The Python SDK is schema-authoring only — it defines GraphQL types, queries, and mutations and exports them to JSON. Observer registration, retry, DLQ, and dispatch are entirely Rust-runtime concerns, reached either through `POST /api/observers` (production) or by embedding `fraiseql-observers` in a host binary (library API).
:::

What changed:

- `@fraiseql.observer(...)` decorator → **gone**. Use `POST /api/observers` to register observers at runtime, or write rows directly to `tb_observer`.
- `actions=[slack("#sales", "..."), email(...), webhook(...)]` Python helpers → **gone**. Use the JSON `actions` array on the observer body with `type: "webhook" | "slack" | "email"` (note: `email` is a stub — FW-22).
- `RetryConfig(max_attempts=5, backoff="exponential", initial_delay_ms=100)` Python class → **gone**. Use the JSON `retry` object on the observer body with the same field names.
- The TOML keys `[observers] backend = "redis"`, `redis_url = "..."`, `nats_url = "..."`, and `[[observers.handlers]]` are **silently ignored** by the binary. They match the `fraiseql-cli compile`-time validation schema, not the running server's runtime schema (FW-15 + FW-23 below).
- `pg_notify('fraiseql_changes', ...)` LISTEN/NOTIFY framing → mostly correct, but the actual mechanism polls `tb_entity_change_log` rather than receiving LISTEN events directly. The default `poll_interval_ms` is 100 ms.

## Known issues

Eleven framework bugs are open against the v2.3.2 observer surface. The four security-class ones are detailed under [Security caveats](#security-caveats); the rest are operational regressions.

| ID | Class | Symptom | Workaround |
|---|---|---|---|
| FW-13 [#340](https://github.com/fraiseql/fraiseql/issues/340) | qol | Observer runtime health / reload routes mounted at root (`/runtime/*`) instead of under `/api/observers/runtime/*`, contrary to other observer routes. | Document the root mount paths explicitly (this page does — see [Route mount paths](#route-mount-paths)). Operators must reach `GET /runtime/health` at root. |
| FW-14 [#341](https://github.com/fraiseql/fraiseql/issues/341) | regression | The `fraiseql-cli` binary has no `observer` subcommand; the standalone `fraiseql-observers` CLI binary returns hard-coded mock JSON for DLQ subcommands. | Use the HTTP API at `/api/observers/dlq` instead of CLI commands. See the [Observer runbook](/operations/observer-runbook) for the `curl`-driven replacement flow. |
| FW-15 [#342](https://github.com/fraiseql/fraiseql/issues/342) | regression | TOML schema is split between `fraiseql-cli` (validation, with `backend`/`handlers`) and `fraiseql-server` (runtime, with `poll_interval_ms`/`pool`); the runtime silently swallows the validation-side fields because `ServerConfig` lacks `deny_unknown_fields`. | Write `fraiseql.toml` against the runtime shape ([documented above](#the-binarys-observers-toml--runtime-knobs-only)). Install observers via `POST /api/observers`. |
| FW-16 [#343](https://github.com/fraiseql/fraiseql/issues/343) | regression | `InMemoryDlq` grows unbounded; the documented `max_dlq_size` cap is exposed only on the library-side `ObserverRuntimeConfig`, not on the binary's `ObserverConfig`. `mark_retry_failed` silently `items.retain()`-removes the failed item — operators lose audit on retry failures. | Monitor `fraiseql_observer_dlq_items` externally; alert and drain via `DELETE /api/observers/dlq/{id}` when depth grows. Do not rely on `max_dlq_size` in TOML. |
| FW-17 [#344](https://github.com/fraiseql/fraiseql/issues/344) | regression | Two concurrent `POST /api/observers/dlq/{id}/retry` calls on the same item both succeed at `dlq.get(id)`, both call `executor.process_event(...)`, both attempt `dlq.remove(id)`. The downstream webhook fires twice. At-least-once contract slips to at-least-twice in the retry race case. | Serialise DLQ retry calls externally (a single operator process invoking the endpoint), or wait for the framework fix before exposing `/api/observers/dlq/{id}/retry` to multiple consumers. |
| FW-18 [#345](https://github.com/fraiseql/fraiseql/issues/345) | security | Webhook payloads are not HMAC-signed; receivers cannot verify origin. See [Security caveats](#fw-18-345--webhook-payloads-are-not-signed-no-hmac). | Terminate webhook subscriptions at an authenticating gateway. |
| FW-19 [#346](https://github.com/fraiseql/fraiseql/issues/346) | security | Webhook URL, headers, and rendered body logged at INFO; PII + bearer-token leak via application logs. See [Security caveats](#fw-19-346--webhook-url-headers-and-body-logged-at-info-pii--bearer-token-leak). | `RUST_LOG=fraiseql_server=warn,fraiseql_observers=warn` in production. |
| FW-20 [#347](https://github.com/fraiseql/fraiseql/issues/347) | security (critical) | `FRAISEQL_OBSERVERS_ALLOW_INSECURE=true` disables every SSRF guard in nine call sites; one-shot WARN at startup is the only signal. See [Security caveats](#fw-20-347--fw-21-348--anonymous-admin-api--ssrf-bypass-env-var-critical-combined). | Never set in any environment with network access to RFC 1918 / link-local addresses. |
| FW-21 [#348](https://github.com/fraiseql/fraiseql/issues/348) | security (critical) | Observer admin API (`POST /api/observers`, `/runtime/reload`, `/dlq/*`) accepts anonymous requests. See [Security caveats](#fw-20-347--fw-21-348--anonymous-admin-api--ssrf-bypass-env-var-critical-combined). | Gate `/api/observers/*` and `/runtime/*` behind an authenticating reverse proxy or VPN-only network. |
| FW-22 [#349](https://github.com/fraiseql/fraiseql/issues/349) | security | `EmailAction` is a stub that returns success without sending email; metrics show 100 % delivery. See [Security caveats](#fw-22-349--emailaction-is-a-stub-that-reports-success). | Route email through a webhook handler to a real email gateway. Do not use `action = "email"` at v2.3.2. |
| FW-23 [#350](https://github.com/fraiseql/fraiseql/issues/350) | regression | Binary ignores `FRAISEQL_OBSERVER_TRANSPORT` and `[observers.transport]`; hard-wired to Postgres LISTEN/NOTIFY even with `observers-nats` feature compiled in. | Embed `fraiseql-observers` in a host binary and call the library API to select the NATS or InMemory transport. |
