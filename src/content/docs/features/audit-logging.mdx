FraiseQL provides comprehensive audit logging for compliance and security monitoring, with multiple backend options.

## Overview

Audit logging tracks:
- GraphQL queries and mutations
- Authentication events (login, logout, token refresh)
- Authorization decisions (access granted/denied)
- Administrative operations
- Data access patterns

## Backends

### PostgreSQL

Store audit logs in the database for querying and retention. Enable via `[security.enterprise]`:

```toml
[security.enterprise]
audit_logging_enabled = true
audit_log_backend     = "postgresql"   # "postgresql" | "syslog" | "file"
```

FraiseQL writes to the `tb_audit_log` table you create in your database (see schema below). Retention and batching are managed by your PostgreSQL maintenance jobs.

### Syslog and File Backends

Set `audit_log_backend` to `"syslog"` or `"file"` to forward audit events to those destinations instead of PostgreSQL. All backends are configured via the same `[security.enterprise]` TOML section — no separate namespace is needed.

## Event Types

### Authentication Events

| Event | Description |
|-------|-------------|
| `auth.login` | User login attempt |
| `auth.login.success` | Successful login |
| `auth.login.failure` | Failed login |
| `auth.logout` | User logout |
| `auth.token.refresh` | Token refreshed |
| `auth.token.expired` | Token expired |
| `auth.session.created` | Session created |
| `auth.session.destroyed` | Session ended |

### Authorization Events

| Event | Description |
|-------|-------------|
| `authz.access.granted` | Access allowed |
| `authz.access.denied` | Access denied |
| `authz.role.required` | Role check performed |
| `authz.scope.required` | Scope check performed |

### Operation Events

| Event | Description |
|-------|-------------|
| `graphql.query` | Query executed |
| `graphql.mutation` | Mutation executed |
| `graphql.subscription` | Subscription started |
| `graphql.error` | Operation error |

Every audit event includes a `transport` field in the `details JSONB` column indicating which transport handled the request: `"graphql"`, `"rest"`, or `"grpc"`. This lets you filter audit logs by transport:

```sql
-- Audit events from REST clients only
SELECT * FROM tb_audit_log
WHERE details->>'transport' = 'rest'
  AND timestamp > NOW() - INTERVAL '1 hour'
ORDER BY timestamp DESC;

-- gRPC mutations in the last 24 hours
SELECT event_type, user_id, details->>'transport' AS transport, timestamp
FROM tb_audit_log
WHERE details->>'transport' = 'grpc'
  AND event_type = 'graphql.mutation'
  AND timestamp > NOW() - INTERVAL '24 hours';
```

### Administrative Events

| Event | Description |
|-------|-------------|
| `admin.schema.reload` | Schema reloaded |
| `admin.cache.clear` | Cache cleared |
| `admin.key.rotate` | Encryption key rotated |

## Configuration Options

The confirmed TOML configuration for audit logging is:

```toml
[security.enterprise]
audit_logging_enabled = true
audit_log_backend     = "postgresql"   # "postgresql" | "syslog" | "file"
audit_retention_days  = 365
```

Log level filtering, event-type filtering, and field redaction are handled in your PostgreSQL `tb_audit_log` table and views — for example, exclude sensitive fields from the audit view, or use PostgreSQL column-level permissions to protect PII fields.

### Field Redaction

Protect sensitive data by not inserting it into the audit table in the first place. The `tb_audit_log` schema below has a `details JSONB` column — write your PostgreSQL audit trigger to omit or hash sensitive fields before inserting:

```sql
-- Hash email for correlation without storing PII
INSERT INTO tb_audit_log (event_type, user_id, details)
VALUES (
    'auth.login.success',
    p_user_id,
    jsonb_build_object(
        'email_hash', encode(sha256(p_email::bytea), 'hex')
        -- never store p_email directly
    )
);
```

## Querying Audit Logs

### PostgreSQL Queries

```sql
-- Recent failed logins
SELECT * FROM tb_audit_log
WHERE event_type = 'auth.login.failure'
  AND timestamp > NOW() - INTERVAL '1 hour'
ORDER BY timestamp DESC;

-- User activity
SELECT event_type, COUNT(*), MAX(timestamp)
FROM tb_audit_log
WHERE user_id = 'user-uuid-here'
  AND timestamp > NOW() - INTERVAL '7 days'
GROUP BY event_type;

-- Access denied events
SELECT * FROM tb_audit_log
WHERE event_type = 'authz.access.denied'
ORDER BY timestamp DESC
LIMIT 100;

-- Mutations by resource
SELECT
    resource,
    COUNT(*) as mutation_count,
    COUNT(DISTINCT user_id) as unique_users
FROM tb_audit_log
WHERE event_type = 'graphql.mutation'
  AND timestamp > NOW() - INTERVAL '24 hours'
GROUP BY resource
ORDER BY mutation_count DESC;
```

### GraphQL Query

Expose audit logs via GraphQL (admin only):

```python
@fraiseql.query(
    sql_source="v_audit_log",
    requires_role="admin"
)
def audit_logs(
    event_type: str | None = None,
    user_id: ID | None = None,
    since: DateTime | None = None,
    limit: int = 100
) -> list[AuditLog]:
    pass
```

## Retention and Cleanup

### Automatic Cleanup

Use a PostgreSQL cron job (via `pg_cron`) or an external scheduler:

```sql
-- Delete audit logs older than 365 days (run as a scheduled job)
DELETE FROM tb_audit_log
WHERE timestamp < NOW() - INTERVAL '365 days';
```

### Manual Cleanup

```sql
-- Manual cleanup
DELETE FROM tb_audit_log
WHERE timestamp < NOW() - INTERVAL '365 days';
```

### Archive Before Delete

Archive to S3 or cold storage using `pg_dump` or your data pipeline before deleting:

```bash
# Export audit logs older than 365 days to JSONB
psql $DATABASE_URL -c "COPY (
    SELECT row_to_json(tb_audit_log)
    FROM tb_audit_log
    WHERE timestamp < NOW() - INTERVAL '365 days'
) TO '/tmp/audit-archive.json'"

# Upload to S3, then delete from DB
aws s3 cp /tmp/audit-archive.json s3://your-bucket/audit/$(date +%Y%m%d).json
psql $DATABASE_URL -c "DELETE FROM tb_audit_log WHERE timestamp < NOW() - INTERVAL '365 days'"
```

## Compliance

Enable audit logging via `[security.enterprise]`, then implement compliance-specific controls at the PostgreSQL layer.

### HIPAA

Log all PHI access events. FraiseQL writes every query and mutation to the audit table when `audit_logging_enabled = true`. Add retention enforcement via `pg_cron`:

```sql
-- Retain PHI access logs for 6 years (HIPAA minimum)
SELECT cron.schedule('hipaa-retention', '0 2 * * *', $$
    DELETE FROM tb_audit_log
    WHERE timestamp < NOW() - INTERVAL '6 years'
      AND event_type LIKE 'graphql.%'
$$);
```

Redact sensitive fields before inserting (see Field Redaction above).

### SOC 2

Enable `audit_logging_enabled = true`. The audit table captures all access control decisions (`authz.access.granted`, `authz.access.denied`) and mutations automatically. No additional configuration required.

### GDPR

For right-to-erasure support, implement a PostgreSQL function that anonymises the `user_id` column in audit rows when a deletion request is processed:

```sql
-- Anonymise audit log entries for deleted users
CREATE FUNCTION fn_anonymise_user_audit(p_user_id UUID) RETURNS VOID AS $$
BEGIN
    UPDATE tb_audit_log
    SET user_id = NULL,
        details = details - 'email' - 'ip_address'
    WHERE user_id = p_user_id;
END;
$$ LANGUAGE plpgsql;
```

## Best Practices

### Log Mutations, Sample Queries

Enable `audit_logging_enabled = true`. FraiseQL logs all mutations, auth events, and access denials. For high-volume query logging, insert a sampling condition in your PostgreSQL audit trigger or view.

### Use Structured Details

The `details JSONB` column in `tb_audit_log` stores event-specific context. FraiseQL populates it automatically for GraphQL operations (operation name, variables, result status). For compliance-specific events you add via PostgreSQL triggers, use structured JSON objects that are easy to query with `->` and `->>` operators — not freeform strings.

### Monitor Audit System Health

```
# Alert on audit write failures
rate(fraiseql_audit_errors_total[5m]) > 0

# Alert on queue backup
fraiseql_audit_queue_size > 1000
```

## Troubleshooting

### Missing Audit Events

1. Check `enabled = true`
2. Verify log level isn't filtering
3. Check filter rules
4. Review backend connectivity

### High Latency

1. Increase `batch_size`
2. Increase `flush_interval_ms`
3. Check database/syslog performance
4. Consider async writes

### Disk Space Issues

1. Enable retention cleanup
2. Reduce retention period
3. Archive to cheaper storage
4. Increase sampling for high-volume events
