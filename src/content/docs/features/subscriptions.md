FraiseQL supports GraphQL subscriptions for real-time updates, with multiple transport options optimized for different use cases.

## Overview

Subscriptions push data to clients when events occur:

```graphql
subscription {
    orderUpdated(customerId: "123") {
        id
        status
        updatedAt
    }
}
```

When an order changes, connected clients receive:

```json
{
    "data": {
        "orderUpdated": {
            "id": "order-456",
            "status": "shipped",
            "updatedAt": "2024-01-15T10:30:00Z"
        }
    }
}
```

## Transports

### WebSocket (graphql-ws)

The default transport for browser clients. Uses the modern graphql-ws protocol.

**Configuration:**

```toml
[subscriptions]
enabled = true
transport = "websocket"
path = "/ws"
ping_interval = 30  # keepalive ping every 30s
connection_timeout = 5  # 5s to send connection_init
```

**Client connection (JavaScript):**

```javascript
import { createClient } from 'graphql-ws';

const client = createClient({
    url: 'wss://api.example.com/ws',
    connectionParams: {
        authToken: 'Bearer ...'
    }
});

// Subscribe
const unsubscribe = client.subscribe(
    {
        query: `subscription { orderUpdated { id status } }`
    },
    {
        next: (data) => console.log('Update:', data),
        error: (err) => console.error('Error:', err),
        complete: () => console.log('Complete')
    }
);
```

### PostgreSQL LISTEN/NOTIFY

For server-to-server communication without WebSocket overhead.

**Configuration:**

```toml
[subscriptions]
enabled = true
transport = "postgres_notify"
channel_prefix = "fraiseql_"
```

**Database trigger:**

```sql
-- Automatic trigger generated by FraiseQL
CREATE FUNCTION notify_order_update() RETURNS trigger AS $$
BEGIN
    PERFORM pg_notify(
        'fraiseql_order_updated',
        json_build_object(
            'id', NEW.id,
            'status', NEW.status,
            'customer_id', NEW.fk_customer
        )::text
    );
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

### Webhook Delivery

Push events to HTTP endpoints. Useful for serverless or third-party integrations.

**Configuration:**

```toml
[subscriptions]
enabled = true
transport = "webhook"

[subscriptions.webhook]
endpoint = "${WEBHOOK_ENDPOINT}"
secret = "${WEBHOOK_SECRET}"
timeout = 10
retry_attempts = 3
retry_delay = 1000
```

**Webhook payload:**

```json
{
    "subscription": "orderUpdated",
    "data": {
        "id": "order-456",
        "status": "shipped"
    },
    "timestamp": "2024-01-15T10:30:00Z",
    "signature": "sha256=..."
}
```

**Signature verification:**

```python
import hmac
import hashlib

def verify_webhook(payload: bytes, signature: str, secret: str) -> bool:
    expected = 'sha256=' + hmac.new(
        secret.encode(),
        payload,
        hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(signature, expected)
```

### Kafka Streaming

For high-throughput event streaming.

**Configuration:**

```toml
[subscriptions]
enabled = true
transport = "kafka"

[subscriptions.kafka]
brokers = ["${KAFKA_BROKERS}"]
topic_prefix = "fraiseql."
client_id = "fraiseql-server"
```

## Defining Subscriptions

### Schema Definition

```python
import fraiseql

@fraiseql.type
class Order:
    id: fraiseql.ID
    status: str
    customer_id: fraiseql.ID
    updated_at: fraiseql.DateTime

@fraiseql.subscription(
    sql_source="tb_order",
    event="UPDATE"
)
def order_updated(customer_id: fraiseql.ID | None = None) -> Order:
    """Subscribe to order updates, optionally filtered by customer."""
    pass
```

### Event Filtering

```python
@fraiseql.subscription(
    sql_source="tb_order",
    event="UPDATE",
    filter="status.changed()"  # Only when status changes
)
def order_status_changed() -> Order:
    pass
```

### Multiple Events

```python
@fraiseql.subscription(
    sql_source="tb_order",
    event=["INSERT", "UPDATE", "DELETE"]
)
def order_changed() -> Order:
    """Subscribe to any order change."""
    pass
```

## WebSocket Protocol

FraiseQL implements the graphql-ws (graphql-transport-ws) protocol.

### Message Flow

```
Client                          Server
  |                               |
  |------- connection_init ------>|
  |<------ connection_ack --------|
  |                               |
  |------- subscribe ------------>|
  |<------ next (data) -----------|
  |<------ next (data) -----------|
  |<------ next (data) -----------|
  |                               |
  |------- complete ------------->|
  |                               |
  |<------ ping ------------------|
  |------- pong ----------------->|
```

### Connection Parameters

Pass authentication via `connectionParams`:

```javascript
const client = createClient({
    url: 'wss://api.example.com/ws',
    connectionParams: {
        authToken: 'Bearer eyJ...',
        tenantId: 'tenant-123'
    }
});
```

Server-side access:

```rust
// In Rust server, connectionParams are available during connection_init
let auth_token = connection_params.get("authToken");
```

## Subscription Filtering

### Client-Side Variables

```graphql
subscription OrderUpdates($customerId: ID!) {
    orderUpdated(customerId: $customerId) {
        id
        status
    }
}
```

```javascript
client.subscribe(
    {
        query: `subscription OrderUpdates($customerId: ID!) {
            orderUpdated(customerId: $customerId) { id status }
        }`,
        variables: { customerId: '123' }
    },
    handlers
);
```

### Server-Side Filtering

FraiseQL filters events before sending to clients:

```python
@fraiseql.subscription(
    sql_source="tb_order",
    event="UPDATE",
    auth_filter="customer_id == context.user_id"  # Only user's orders
)
def my_order_updated() -> Order:
    pass
```

## Error Handling

### Connection Errors

```javascript
const client = createClient({
    url: 'wss://api.example.com/ws',
    retryAttempts: 5,
    retryWait: async (retries) => {
        // Exponential backoff
        await new Promise(r => setTimeout(r, 1000 * Math.pow(2, retries)));
    },
    on: {
        error: (err) => console.error('Connection error:', err),
        closed: () => console.log('Connection closed')
    }
});
```

### Subscription Errors

```javascript
client.subscribe(
    { query: '...' },
    {
        error: (errors) => {
            // GraphQL errors during subscription
            console.error('Subscription errors:', errors);
        }
    }
);
```

FraiseQL batches notifications within a time window:

```toml

batch_interval_ms = 50  # Batch events within 50ms
max_batch_size = 100    # Max events per batch
```

## Security

### Authentication

WebSocket connections require authentication:

```toml
[subscriptions.auth]
required = true
token_header = "Authorization"
validate_on_subscribe = true
```

### Rate Limiting

```toml
[subscriptions.rate_limit]
connections_per_ip = 100
subscriptions_per_connection = 50
events_per_second = 1000
```

### Subscription Depth

Prevent expensive subscriptions:

```toml
[subscriptions]
max_selection_depth = 5
max_selected_fields = 50
```

## Best Practices

### Use Specific Subscriptions

```graphql
# Good - filtered
subscription {
    orderUpdated(customerId: "123") { id status }
}

# Avoid - unfiltered (receives all orders)
subscription {
    orderUpdated { id status }
}
```

### Handle Reconnection

```javascript
const client = createClient({
    url: 'wss://api.example.com/ws',
    keepAlive: 10000,  // Ping every 10s
    retryAttempts: Infinity,
    shouldRetry: () => true
});
```

### Clean Up Subscriptions

```javascript
// Store unsubscribe function
const unsubscribe = client.subscribe(...);

// Clean up on unmount
useEffect(() => {
    return () => unsubscribe();
}, []);
```

## Troubleshooting

### Connection Drops

1. Check `ping_interval` matches client expectations
2. Verify no proxy timeouts (configure proxy keepalive)
3. Check server logs for connection errors

### Missing Events

1. Verify subscription filter matches data
2. Check auth filter isn't too restrictive
3. Confirm database triggers are active

### High Latency

1. Enable batching for high-frequency events
2. Consider Kafka for extreme throughput
3. Check database NOTIFY backpressure
