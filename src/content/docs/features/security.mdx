This page is the security cluster's hub. It covers the two subsystems that uniquely live here — **JWT verification at the request edge** and **field-level RBAC (including the runtime RBAC management API)** — and cross-links every other security topic to its dedicated page. The authoritative source for the full authentication surface (`[auth]` OIDC validator, `[auth_hs256]`, PKCE OAuth2, token revocation, `/auth/me`, framework caveats) is [Building → Authentication](/building/authentication); read it before exposing any auth route to the public internet.

## Security caveats

Six framework issues are open against the v2.3.2 authentication surface; one is critical, three are security regressions, two are operator footguns. The full caveat block — including mitigations and `## Known issues` — lives on [Building → Authentication](/building/authentication#security-caveats); the table below names the symptom and points at the matching caveat anchor. **Do not expose `/auth/revoke*` or the RBAC management API to the public internet without an authenticating reverse proxy.**

| ID | Symptom | Where documented |
|---|---|---|
| FW-24 [#356](https://github.com/fraiseql/fraiseql/issues/356) | `[security.rate_limiting] failed_login_max_attempts` / `failed_login_lockout_secs` are silently dropped by the server runtime. Brute-force lockout is a no-op. | [Auth caveat 6](/building/authentication#6-fw-24-356--brute-force-protection-silently-dropped-security) |
| FW-25 [#357](https://github.com/fraiseql/fraiseql/issues/357) | `[security.token_revocation] backend = "postgres"` silently downgrades to in-memory. | [Auth caveat 7](/building/authentication#7-fw-25-357--token-revocation-backend--postgres-silently-downgrades-regression) |
| FW-26 [#358](https://github.com/fraiseql/fraiseql/issues/358) (**critical**) | `POST /auth/revoke` and `POST /auth/revoke-all` are mounted with no auth middleware. | [Auth caveat 2](/building/authentication#2-fw-26-358--authrevoke-and-authrevoke-all-are-unauthenticated-critical) |
| FW-27 [#359](https://github.com/fraiseql/fraiseql/issues/359) | HS256 boot without `audience` succeeds; presented tokens are not audience-checked. | [Auth caveat 3](/building/authentication#3-fw-27-359--hs256-audience-is-not-enforced-by-the-framework-security) |
| FW-28 [#360](https://github.com/fraiseql/fraiseql/issues/360) | PKCE with `state_encryption` disabled warns once and continues; `state` is unencrypted. | [Auth caveat 4](/building/authentication#4-fw-28-360--pkce-warns-but-continues-without-state-encryption-security) |
| FW-29 [#361](https://github.com/fraiseql/fraiseql/issues/361) | JWKS cache holds rotated-out keys for up to `jwks_cache_ttl_secs`; `detect_key_rotation` warns but does not flush. | [Auth caveat 5](/building/authentication#5-fw-29-361--jwks-hot-rotate-window-equals-the-cache-ttl-security) |

The observer admin surface (FW-21 [#348](https://github.com/fraiseql/fraiseql/issues/348)) is security-adjacent and shares the same "unauthenticated admin route" class as FW-26; see [Observers](/features/observers#security-caveats) for the dedicated treatment.

## JWT verification at the request edge

FraiseQL verifies a JWT on every incoming request through `oidc_auth_middleware`. The middleware prefers an `Authorization: Bearer <token>` header and falls back to the `__Host-access_token` HttpOnly cookie set by the PKCE callback. JWTs are validated against the IdP's JWKS (RS256 by default) or against a shared symmetric secret (HS256) for testing.

{/* source: crates/fraiseql-server/src/middleware/oidc_auth.rs:L48-L62 — extract_access_token_cookie */}
{/* source: crates/fraiseql-server/src/middleware/oidc_auth.rs:L83-L97 — oidc_auth_middleware doc comment: Bearer header → cookie fallback */}

The validator is configured via `[auth]` (OIDC) or `[auth_hs256]` (testing) in `fraiseql.toml`. Both sections are direct-TOML on `ServerConfig` — the binary auto-wires them at startup without a `fraiseql-cli compile` step.

{/* source: crates/fraiseql-server/src/server_config/mod.rs:L293-L307 — auth: Option<OidcConfig> */}
{/* source: crates/fraiseql-server/src/server_config/hs256.rs:L11-L39 — Hs256Config struct + mutual exclusion docstring */}

```toml title="fraiseql.toml — production OIDC"
[auth]
issuer = "https://your-tenant.auth0.com/"
audience = "https://api.example.com"     # SECURITY CRITICAL — boot fails without it
# allowed_algorithms = ["RS256"]         # default; do not widen (algorithm-confusion risk)
# jwks_cache_ttl_secs = 300              # default; tighten only if your IdP supports fast rotation
```

`OidcConfig::validate()` refuses to boot when both `audience` and `additional_audiences` are empty — this is the framework's mandatory audience guard against token-confusion attacks. HS256 has no equivalent guard at v2.3.2 (FW-27); set `[auth_hs256] audience` explicitly even though the framework allows you to omit it.

{/* source: crates/fraiseql-core/src/security/oidc/providers.rs:L290-L320 — validate() refuses boot when audience + additional_audiences both empty */}
{/* source: crates/fraiseql-core/src/security/oidc/providers.rs:L45-L73 — audience field SECURITY CRITICAL doc comment */}

### What the validator returns

When the JWT validates, the middleware constructs a `SecurityContext` exposing `user_id` (from `sub`), `email`, `display_name` (both normalised from common claim shapes), `expires_at`, `scopes` (parsed from the configured `scope_claim`, default `scope`), `roles`, and `extra_claims`. RLS policies access these via `jwt:sub`, `jwt:email`, `jwt:name`, and `jwt:display_name` session-variable mappings. Custom claims are injected per-query via `inject={"sql_param": "jwt:<claim>"}` (see [Server-side injection](/features/server-side-injection)).

{/* source: crates/fraiseql-core/src/security/security_context.rs:L115-L127 — email + display_name fields */}
{/* source: crates/fraiseql-core/src/security/oidc/providers.rs:L115-L120 — scope_claim default */}

### Algorithm allowlist — keep it `["RS256"]`

`OidcConfig::default_algorithms()` returns `vec!["RS256".to_string()]`. The validator enforces this list at decode time and rejects tokens signed with disallowed algorithms (`SecurityError::InvalidTokenAlgorithm`). Widening the allowlist to include `HS256` on a JWKS-only OIDC deployment opens the classic algorithm-confusion attack — keep the default unless you have a documented reason and a second-layer mitigation.

{/* source: crates/fraiseql-core/src/security/oidc/providers.rs:L152-L154 — default_algorithms() returns ["RS256"] */}
{/* source: crates/fraiseql-core/src/security/oidc/token.rs:L355-L368 — get_algorithm enforces allowlist; returns InvalidTokenAlgorithm */}

### Anonymous baseline

When neither `[auth]` nor `[auth_hs256]` is configured, the binary boots, mounts `/graphql` and `/studio`, and serves every request as anonymous. `*_require_auth` flags default `true` but only enforce when a validator is installed; without one, the require-auth check is a no-op. Always configure `[auth]` or `[auth_hs256]` before exposing the binary to any non-localhost network.

{/* source: crates/fraiseql-server/src/server/builder.rs:L308-L323 — OIDC validator built only when config.auth is Some */}
{/* source: crates/fraiseql-server/src/server/builder.rs:L19-L39 — build_hs256_auth returns Ok(None) when [auth_hs256] absent */}

## Field-level RBAC

FraiseQL enforces role and scope requirements directly on type fields and on whole queries and mutations. The shapes are declared in the Python SDK at authoring time and baked into the compiled schema; the Rust runtime enforces them at execution time.

### Requiring scopes on fields

```python
from typing import Annotated
from decimal import Decimal
import fraiseql
from fraiseql.scalars import ID

@fraiseql.type
class User:
    id: ID
    name: str
    email: str

    # Protected field — requires scope
    salary: Annotated[Decimal, fraiseql.field(requires_scope="hr:read_salary")]

    # Multiple required scopes (all must be present)
    ssn: Annotated[str, fraiseql.field(requires_scope="hr:read_pii admin:view")]
```

{/* source: crates/fraiseql-cli/src/schema/converter/types.rs:L182 — requires_scope flows from intermediate to converted FieldDefinition */}

When a client requests a protected field without the required scope, FraiseQL returns `null` for the field and includes an error entry:

```json
{
  "data": {
    "user": { "name": "John", "email": "john@example.com", "salary": null }
  },
  "errors": [{
    "message": "Forbidden: missing scope hr:read_salary",
    "path": ["user", "salary"],
    "extensions": { "code": "FORBIDDEN", "statusCode": 403 }
  }]
}
```

### Requiring roles on types, queries, and mutations

```python
@fraiseql.type(requires_role="admin")
class AdminDashboard:
    total_users: int
    revenue: float

@fraiseql.query(
    sql_source="v_user",
    requires_role="admin",
)
def all_users() -> list[User]:
    """Only admins can list all users."""

@fraiseql.mutation(
    sql_source="fn_delete_user",
    requires_scope="admin:delete_user",
)
def delete_user(id: ID) -> bool:
    """Requires admin:delete_user scope."""
```

{/* source: crates/fraiseql-cli/src/schema/converter/queries.rs:L158 — requires_role flows from intermediate to QueryDefinition */}
{/* source: crates/fraiseql-cli/src/schema/converter/types.rs:L40 — requires_role on type-level fields */}

### Authorization policies (TOML-driven)

For query-scoped and row-level authorization that does not fit a single annotation, FraiseQL accepts TOML-configured policies. The shapes live in the CLI's `[fraiseql.security.*]` block and are compiled into the schema; the server reads them from `schema.compiled.json` at startup.

{/* source: crates/fraiseql-cli/src/config/toml_schema/security.rs:L7-L34 — SecuritySettings with rules + policies + field_auth */}

```toml title="fraiseql.toml — TOML-driven authorization (compile-step)"
[fraiseql.security]
default_policy = "authenticated"   # "authenticated" | "public"

[[fraiseql.security.rules]]
name              = "owner_only"
rule              = "user.id == object.owner_id"
description       = "User can only access their own data"
cacheable         = true
cache_ttl_seconds = 300

[[fraiseql.security.policies]]
name              = "admin_only"
type              = "rbac"          # "rbac" | "abac" | "custom" | "hybrid"
roles             = ["admin"]
strategy          = "any"           # "any" | "all" | "exactly"
cache_ttl_seconds = 600

[[fraiseql.security.field_auth]]
type_name  = "User"
field_name = "email"
policy     = "admin_only"
```

The fields above are the verified accepted shape at the frozen SHA. Adding unknown keys at the CLI compile step raises a hard "unknown field" error (the CLI's TOML schema uses `deny_unknown_fields`).

{/* source: crates/fraiseql-cli/src/config/toml_schema/security.rs:L9 — SecuritySettings serde(default, deny_unknown_fields) */}
{/* source: crates/fraiseql-cli/src/config/toml_schema/security.rs:L60-L70 — AuthorizationRule deny_unknown_fields */}
{/* source: crates/fraiseql-cli/src/config/toml_schema/security.rs:L73-L80 — AuthorizationPolicy deny_unknown_fields */}

## RBAC management API

When `admin_token` is set on `ServerConfig`, the binary mounts a small REST router for runtime role / permission / user-role-assignment management. The router is at `/api/` (not `/api/rbac/` — that prefix appeared in earlier docs and does not match the framework). Every route is gated by a bearer-token middleware that compares the request's `Authorization: Bearer <admin_token>` against the configured value.

{/* source: crates/fraiseql-server/src/server/routing/extensions.rs:L160-L174 — RBAC router mounted under admin_token guard; route_layer applies bearer_auth_middleware */}
{/* source: crates/fraiseql-server/src/api/rbac_management.rs:L117-L135 — rbac_management_router routes */}

### Routes

```
POST   /api/roles                                Create role
GET    /api/roles                                List roles
GET    /api/roles/{role_id}                      Get role
PUT    /api/roles/{role_id}                      Update role
DELETE /api/roles/{role_id}                      Delete role

POST   /api/permissions                          Create permission
GET    /api/permissions                          List permissions
GET    /api/permissions/{permission_id}          Get permission
DELETE /api/permissions/{permission_id}          Delete permission

POST   /api/user-roles                           Assign role to user
GET    /api/user-roles                           List user-role assignments
DELETE /api/user-roles/{user_id}/{role_id}       Revoke role from user

GET    /api/audit/permissions                    Query permission access audit logs
```

{/* source: crates/fraiseql-server/src/api/rbac_management.rs:L117-L135 — routes list */}

### `admin_token` gating

The router is mounted only when `admin_token` is set; without it, the framework logs a startup error (`RBAC Management API disabled — admin_token is not set`) and the routes are never registered (requests return 404). When `admin_token` is set, the bearer-token middleware enforces the token on every request; misses are counted against `admin_auth_max_failures` (default applies) for the lockout circuit.

{/* source: crates/fraiseql-server/src/server/routing/extensions.rs:L160-L179 — admin_token Some/None branches with route mounting and error log */}
{/* source: crates/fraiseql-server/src/server_config/mod.rs:L194-L207 — admin_api_enabled + admin_token doc comments */}

<Aside type="caution" title="admin_token grants destructive operations">
The RBAC routes share the bearer-token gate with `reload-schema`, `cache/clear`, and other admin operations on the server. Use a 32+ character random token, store it only as `${FRAISEQL_ADMIN_TOKEN}` (env-var substitution from TOML), and gate the routes at a reverse proxy in addition to the bearer-token check. If you need read-only access for monitoring tools, set `admin_readonly_token` separately to a different token and restrict `admin_token` to write operations.

```toml
[server]
admin_api_enabled = true
admin_token = "${FRAISEQL_ADMIN_TOKEN}"
# admin_readonly_token = "${FRAISEQL_ADMIN_RO_TOKEN}"  # optional split
```
</Aside>

{/* source: crates/fraiseql-server/src/server_config/mod.rs:L209-L220 — admin_readonly_token doc comment */}

### Quick mention — error sanitization, query complexity, CORS

These three controls are covered for completeness because they often surface in security reviews; their canonical home is the [TOML configuration reference](/reference/toml-config).

- **Error sanitization** — `[fraiseql.security.error_sanitization]` is disabled by default. Enable it in production so `DatabaseError` and `InternalServerError` codes are replaced with a generic message; `VALIDATION_ERROR`, `NOT_FOUND`, `FORBIDDEN`, `UNAUTHENTICATED`, and `RATE_LIMITED` are never sanitized because clients need them to act on errors.

  {/* source: crates/fraiseql-cli/src/config/toml_schema/security.rs:L42-L52 — error_sanitization: Option<ErrorSanitizationTomlConfig> */}

- **Query complexity** — `[fraiseql.validation]` caps query depth (`max_query_depth`, default 10) and aliases (`max_aliases`, default 30). The 1,000-variable-per-request limit is a hard runtime constant; requests exceeding it are rejected with `VALIDATION_ERROR` before execution.

  {/* source: crates/fraiseql-cli/src/config/toml_schema/server_settings.rs:L49 — ValidationConfig deny_unknown_fields */}

- **CORS** — `[server.cors]` on `ServerConfig`, not `[security.cors]`. Never use wildcard origins when `credentials = true`.

  {/* source: crates/fraiseql-server/src/server_config/mod.rs:L41 — ServerConfig struct (cors lives on server, not security) */}

## Authorization error responses

When a client lacks the required role or scope, FraiseQL returns a structured GraphQL error. The codes are stable; clients should switch on `extensions.code`, not on the message.

```json title="Missing authentication"
{
  "errors": [{
    "message": "Authentication required",
    "extensions": { "code": "UNAUTHENTICATED", "statusCode": 401 }
  }]
}
```

```json title="Missing role / scope"
{
  "errors": [{
    "message": "Access denied: role 'viewer' cannot access field 'User.email'",
    "path": ["user", "email"],
    "extensions": {
      "code": "FORBIDDEN",
      "requiredRole": "admin",
      "statusCode": 403
    }
  }]
}
```

## Testing the security path

<Steps>

1. **Field-level scope enforcement:**

   ```bash
   curl -X POST http://localhost:8080/graphql \
     -H "Authorization: Bearer $USER_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"query":"{ user(id:\"user-123\") { name salary } }"}'
   ```

   Expected: `salary` returns `null` with a `FORBIDDEN` error entry.

2. **Type-level role enforcement:**

   ```bash
   curl -X POST http://localhost:8080/graphql \
     -H "Authorization: Bearer $USER_TOKEN" \
     -H "Content-Type: application/json" \
     -d '{"query":"{ allUsers { id name } }"}'
   ```

   Expected: `FORBIDDEN` error, no data.

3. **RBAC management API auth:**

   ```bash
   # Without admin_token — 401
   curl -fsS http://localhost:8080/api/roles
   # With admin_token — 200
   curl -fsS http://localhost:8080/api/roles \
     -H "Authorization: Bearer $FRAISEQL_ADMIN_TOKEN"
   ```

</Steps>
