FraiseQL supports response caching for GraphQL queries. Caching can reduce database load dramatically for read-heavy workloads.

<Aside type="caution">
**`[caching]` TOML configuration is unverified.** This namespace is not listed as a valid `fraiseql.toml` section (valid sections are documented in the [TOML Config Reference](/reference/toml-config)). The caching behaviour, TTL concepts, and invalidation model described below are accurate. The exact TOML key and field names are subject to change.
</Aside>

<Aside type="note">
Caching is disabled by default. Every GraphQL request goes directly to the database until caching is explicitly enabled.
</Aside>

## Cache Backends

FraiseQL supports three caching backends:

| Backend | Use Case | Persistence |
|---------|----------|-------------|
| `redis` | Production, multi-instance | Survives restarts, shared across instances |
| `memory` | Development, single-instance | Per-process, lost on restart |
| `postgresql` | Fallback when Redis unavailable | Stored in database table |

<Aside type="caution">
The `memory` backend is not shared between instances. If you run multiple FraiseQL replicas behind a load balancer, each instance maintains its own cache, and mutations on one instance will not invalidate cache entries on the others. Use `redis` for multi-instance deployments.
</Aside>

## Redis Cache Setup

<Steps>

1. **Start a Redis instance**

   ```bash
   docker run -d --name redis -p 6379:6379 redis:7-alpine
   ```

2. **Configure the `[caching]` block**

   ```toml
   [caching]
   enabled = true
   backend = "redis"
   redis_url = "redis://localhost:6379"
   ```

3. **Start FraiseQL**

   ```bash
   fraiseql run
   ```

   On startup, FraiseQL connects to Redis and logs confirmation that caching is active.

</Steps>

## In-Memory Cache Setup

For local development or single-instance deployments:

```toml
[caching]
enabled = true
backend = "memory"
```

The in-memory backend requires no external services but:
- Cache is lost when FraiseQL restarts
- Each instance has independent cache in multi-instance setups
- Limited by available heap memory

## Cache Rules and TTL

Cache rules define which queries to cache and for how long. Rules use glob patterns to match query names.

```toml
[caching]
enabled = true
backend = "redis"
redis_url = "redis://localhost:6379"

[[caching.rules]]
query = "posts*"           # Cache all queries starting with "posts"
ttl_seconds = 300        # Cache for 5 minutes
invalidation_triggers = ["Post", "Comment"]  # Invalidate on these mutations

[[caching.rules]]
query = "user*"
ttl_seconds = 60
invalidation_triggers = ["User"]

[[caching.rules]]
query = "config*"        # Static configuration queries
ttl_seconds = 3600       # Cache for 1 hour
invalidation_triggers = []  # Never invalidate (manual only)
```

### Rule Structure

| Field | Type | Description |
|-------|------|-------------|
| `query` | string | Glob pattern for query names (e.g., `posts*`, `*List`, `userById`) |
| `ttl_seconds` | integer | Time-to-live in seconds |
| `invalidation_triggers` | array | Entity names that invalidate this cache when mutated |

### Pattern Matching

- `posts*` matches `posts`, `postsByCategory`, `postsCount`
- `*List` matches `userList`, `productList`
- `userById` matches exactly `userById`

### TTL Guidelines

| Data type | Recommended TTL | Rationale |
|-----------|-----------------|-----------|
| Public static content (site config, feature flags) | 3600 s (1 hour) | Changes rarely, safe to cache long |
| Product listings, post feeds | 300 s (5 min) | Moderate freshness requirement |
| User-specific data | 60 s (1 min) | May change across sessions |
| Real-time or financial data | No caching | Staleness is not acceptable |

## Cache Invalidation

FraiseQL invalidates cache entries automatically when mutations change data. When a mutation runs, FraiseQL checks the `invalidation_triggers` of each cache rule and purges matching entries.

### Automatic Invalidation

Given this configuration:

```toml
[[caching.rules]]
query = "posts*"
ttl_seconds = 300
invalidation_triggers = ["Post", "Comment"]
```

When a `Post` or `Comment` mutation executes:
1. FraiseQL detects the entity change
2. Finds all cache rules with that entity in `invalidation_triggers`
3. Purges cache entries matching those rules
4. Subsequent queries fetch fresh data from the database

### Manual Invalidation

In some cases, you may need to invalidate cache manually (e.g., administrative changes):

<Tabs syncKey="language">
<TabItem label="Python">

```python
from fraiseql import cache

# Invalidate all entries matching a pattern
await cache.invalidate_pattern("posts:*")

# Invalidate specific entity cache
await cache.invalidate_entity("Post", post_id="123")
```

</TabItem>
<TabItem label="TypeScript">

```typescript
import { cache } from 'fraiseql';

// Invalidate all entries matching a pattern
await cache.invalidatePattern('posts:*');

// Invalidate specific entity cache
await cache.invalidateEntity('Post', { postId: '123' });
```

</TabItem>
</Tabs>

## Environment Variable Overrides

All caching settings can be overridden via environment variables:

| Variable | Overrides |
|----------|-----------|
| `FRAISEQL_CACHING_ENABLED` | `caching.enabled` |
| `FRAISEQL_CACHING_BACKEND` | `caching.backend` |
| `FRAISEQL_REDIS_URL` | `caching.redis_url` |

Example:

```bash
export FRAISEQL_CACHING_ENABLED=true
export FRAISEQL_REDIS_URL=redis://prod-redis:6379
fraiseql run
```

## Cache Metrics

FraiseQL exposes Prometheus metrics for cache performance at the `/metrics` endpoint:

| Metric | Type | Description |
|--------|------|-------------|
| `fraiseql_cache_hits_total` | Counter | Total cache hits across all queries |
| `fraiseql_cache_misses_total` | Counter | Total cache misses |
| `fraiseql_cache_invalidations_total` | Counter | Total cache entries invalidated |

### Useful PromQL Expressions

Cache hit rate over a 5-minute window:

```txt
sum(rate(fraiseql_cache_hits_total[5m]))
  /
(
  sum(rate(fraiseql_cache_hits_total[5m]))
  + sum(rate(fraiseql_cache_misses_total[5m]))
)
```

A healthy read-heavy API typically achieves a hit rate above 80%. Values below 50% indicate that:
- TTLs are too short relative to request frequency
- Mutations are invalidating entries too aggressively
- The query patterns don't match the cache rules

Invalidation rate relative to hits:

```txt
rate(fraiseql_cache_invalidations_total[5m])
  /
rate(fraiseql_cache_hits_total[5m])
```

If this ratio exceeds 0.1 (10%), review your invalidation triggers — they may be too broad.

## Troubleshooting

**"Caching enabled but no cache hits"**

Verify caching is active and rules match your queries:

```bash
# Check caching is enabled
grep -A5 '\[caching\]' fraiseql.toml

# Verify rules match query names
# If your query is named `getPosts`, the rule should be `query = "getPosts*"` or `query = "*Posts*"`
```

**"Cache invalidation not working"**

1. Check that the mutation returns the correct entity type name
2. Verify `invalidation_triggers` includes that exact entity name (case-sensitive)
3. Ensure the entity name in the rule matches the name in your schema

**"Out of memory with in-memory cache"**

The in-memory backend has no automatic eviction. For production use cases, switch to Redis:

```toml
[caching]
backend = "redis"
redis_url = "redis://localhost:6379"
```

**"Redis connection errors"**

Verify Redis is accessible:

```bash
redis-cli -h localhost -p 6379 ping
# Should return: PONG
```

Check the connection URL format:

```toml
# Standard format
redis_url = "redis://localhost:6379"

# With password
redis_url = "redis://:password@localhost:6379"

# With database number
redis_url = "redis://localhost:6379/0"
```

## Best Practices

**Use specific invalidation triggers.** Broad triggers (e.g., invalidating all `posts*` on any `User` change) reduce cache effectiveness:

```toml
# Good: Only invalidate when Post or Comment changes
invalidation_triggers = ["Post", "Comment"]

# Avoid: Invalidating on unrelated entity changes
invalidation_triggers = ["Post", "User", "Config", "Settings"]
```

**Choose TTLs based on data change frequency.** Static data can cache longer; frequently changing data needs shorter TTLs or no caching.

**Use Redis for production.** The in-memory backend is convenient for development but unsuitable for production multi-instance deployments.

**Monitor hit rates.** Alert when cache hit rate drops below your threshold (typically 70-80%).

**Test invalidation.** After implementing caching, verify that mutations correctly invalidate cache entries by:
1. Executing a cached query (should hit cache)
2. Running a mutation that should invalidate it
3. Executing the same query again (should hit database, not cache)

## Limitations

Current caching implementation has these limitations:

1. **No cache key customization**: Cache keys are automatically derived from query name and arguments. You cannot customize the key pattern.
2. **No per-query cache headers**: Unlike some GraphQL servers, FraiseQL does not add `X-Cache-Status` headers to responses.
3. **No cache warming**: There is no built-in mechanism to pre-populate the cache.
4. **PostgreSQL backend limitations**: The PostgreSQL cache backend is slower than Redis and not recommended for high-throughput scenarios.
