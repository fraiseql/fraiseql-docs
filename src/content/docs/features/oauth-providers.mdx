This page covers per-provider OIDC configuration. The validator surface itself — `OidcConfig`, audience enforcement, JWKS, algorithm allowlist, `__Host-access_token` cookie, PKCE flow, token revocation, and the 11 security caveats including the six open framework issues — lives on [Building → Authentication](/building/authentication). Read that page first if you have not already.

## Supported providers

The OIDC validator works with any compliant provider. The framework ships convenience constructors for the five most common ones; the generic OIDC path covers everything else.

| Provider | Protocol | Constructor at frozen SHA |
|----------|----------|----------------------------|
| Auth0 | OIDC | `OidcConfig::auth0(domain, audience)` |
| Keycloak | OIDC | `OidcConfig::keycloak(base_url, realm, client_id)` |
| Okta | OIDC | `OidcConfig::okta(domain, audience)` |
| AWS Cognito | OIDC | `OidcConfig::cognito(region, user_pool_id, client_id)` |
| Microsoft Entra ID (Azure AD) | OIDC | `OidcConfig::azure_ad(tenant_id, client_id)` |
| Any compliant provider | OIDC | manual `[auth]` TOML or `OidcConfig::default()` |

{/* source: crates/fraiseql-core/src/security/oidc/providers.rs:L172-L262 — per-provider constructors */}

The constructors populate `OidcConfig` with the right `issuer` URL pattern and set `audience` from the supplied client ID / API identifier; every other field falls back to `OidcConfig::default()`. The configuration is direct-TOML on `ServerConfig.auth: Option<OidcConfig>` — no compile step required.

{/* source: crates/fraiseql-server/src/server_config/mod.rs:L293-L307 — ServerConfig.auth: Option<OidcConfig> with TOML doc example */}

## Configuration

`[auth]` is a valid `fraiseql.toml` section, direct-wired by the binary. The validator refuses to boot when `audience` and `additional_audiences` are both empty — this is the framework's mandatory mitigation for token-confusion attacks.

{/* source: crates/fraiseql-core/src/security/oidc/providers.rs:L290-L320 — validate() refuses boot when audience + additional_audiences both empty */}

```toml title="fraiseql.toml — generic OIDC"
[auth]
issuer = "https://your-provider.example.com/"      # MUST match the iss claim exactly
audience = "https://api.example.com"               # SECURITY CRITICAL — boot fails without it
# additional_audiences = ["legacy-api-id"]         # optional secondary audiences
# allowed_algorithms = ["RS256"]                   # default; do not widen (see Authentication caveat 10)
# jwks_cache_ttl_secs = 300                        # default
# clock_skew_secs = 60                             # default; hard-capped at 300 regardless of value
# scope_claim = "scope"                            # space-separated string; some IdPs use "scp" or "permissions"
# required = true                                  # default — reject requests without a valid token
```

{/* source: crates/fraiseql-core/src/security/oidc/providers.rs:L50-L143 — OidcConfig fields with defaults */}

The OIDC client secret is **never** stored in `fraiseql.toml` for the validator path — the validator only verifies inbound tokens against the IdP's JWKS, which is a public document. The client secret is only needed for the PKCE flow (browser login → callback → token exchange); for that, set `[fraiseql.security.pkce]` and source the secret from `${OIDC_CLIENT_SECRET}` at compile time. See [Authentication → PKCE](/building/authentication#pkce-oauth2-flow).

## Authentication endpoints

The OIDC validator gates every request through `oidc_auth_middleware` (Bearer header preferred, `__Host-access_token` cookie as fallback). The PKCE OAuth2 flow — required for browser logins — mounts two additional endpoints when `[fraiseql.security.pkce]` is configured in the compiled schema.

| Endpoint | Method | Purpose | Mounted when |
|----------|--------|---------|--------------|
| `/auth/start` | GET | Begins the PKCE flow; redirects to the IdP with a generated challenge | `[fraiseql.security.pkce]` + OIDC client identity configured |
| `/auth/callback` | GET | Consumes the verifier; exchanges code at the IdP; sets `__Host-access_token` cookie | `[fraiseql.security.pkce]` + OIDC client identity configured |
| `/auth/me` | GET | Returns the validated session's identity claims | `[auth.me] enabled = true` + OIDC validator present |
| `/auth/revoke` | POST | Revoke a single token by `jti` (**unauthenticated at v2.3.2 — FW-26 #358**) | `[fraiseql.security.token_revocation]` configured |
| `/auth/revoke-all` | POST | Revoke all tokens for a `sub` (**unauthenticated at v2.3.2 — FW-26 #358**) | `[fraiseql.security.token_revocation]` configured |

{/* source: crates/fraiseql-server/src/server/routing/auth.rs:L24-L114 — mount_auth_routes; all five routes shown above */}

<Aside type="caution" title="FW-26 [#358](https://github.com/fraiseql/fraiseql/issues/358) — `/auth/revoke*` is unauthenticated">
`POST /auth/revoke` and `POST /auth/revoke-all` are mounted without any auth middleware at v2.3.2. The sibling `/auth/me` route in the same `mount_auth_routes` function DOES apply `oidc_auth_middleware`, but the revocation routes do not. The `revoke_token` handler decodes the supplied token via `jsonwebtoken::dangerous::insecure_decode` — no signature check. `revoke_all_tokens` accepts a body `{"sub": "..."}` with no token at all.

**Gate `/auth/revoke*` behind an authenticating reverse proxy, or omit `[fraiseql.security.token_revocation]` from the compiled schema entirely** — the routes are not mounted when the manager is absent.

See [Authentication → caveat 2](/building/authentication#2-fw-26-358--authrevoke-and-authrevoke-all-are-unauthenticated-critical) for the full repro and mitigation.
</Aside>

The PKCE flow is documented in detail at [Authentication → PKCE OAuth2 flow](/building/authentication#pkce-oauth2-flow). Other auth extensions (`/auth/v1/authorize` social login, `/auth/v1/signup` anonymous signup, `/auth/v1/mfa/*` TOTP MFA) mount under their own configuration sections; the same `mount_auth_routes` function gates them.

{/* source: crates/fraiseql-server/src/server/routing/auth.rs:L47-L80 — social_login / anon_signup / mfa branches */}

## Authentication flow

The PKCE Authorization Code flow is the recommended browser-login pattern.

<Steps>
1. The client opens `GET /auth/start?...` in a browser. FraiseQL generates a PKCE code verifier and challenge, stores the state server-side in `PkceStateStore`, and redirects the browser to the IdP's authorization endpoint.
2. The user authenticates at the IdP and approves the consent screen. The IdP redirects the browser back to `GET /auth/callback?code=<authcode>&state=<state>`.
3. FraiseQL retrieves the verifier from `PkceStateStore`, exchanges the authorization code with the IdP's token endpoint, and validates the returned access token against the configured `[auth]` audience and algorithm allowlist.
4. The callback sets the `__Host-access_token="<jwt>"; Path=/; HttpOnly; Secure; SameSite=Strict; Max-Age=<expires_in or 300>` cookie and redirects to the configured post-login URI.
5. Subsequent requests carry the cookie automatically (or use `Authorization: Bearer <jwt>` for non-browser clients); `oidc_auth_middleware` validates on every request.
</Steps>

{/* source: crates/fraiseql-server/src/routes/auth.rs:L114 — pub async fn auth_start */}
{/* source: crates/fraiseql-server/src/routes/auth.rs:L168 — pub async fn auth_callback */}
{/* source: crates/fraiseql-server/src/routes/auth.rs:L220-L246 — cookie format hard-coded */}

## JWT claims in queries

FraiseQL is a compiled engine. The Python SDK declares schema at compile time; there is no runtime resolver function with a `Context` object. JWT claims are surfaced to SQL via the `inject={"sql_param": "jwt:<claim>"}` decorator argument — the Rust runtime reads the verified claim from the `SecurityContext` and binds it as a SQL parameter, never as a GraphQL input.

```python
import fraiseql
from fraiseql.scalars import ID

@fraiseql.type
class User:
    id: ID
    email: str
    name: str

# The authenticated user's own profile — user_id injected from jwt:sub
@fraiseql.query(
    sql_source="v_user",
    inject={"user_id": "jwt:sub"},
)
def me() -> User | None:
    """Get the currently authenticated user."""

# Tenant-scoped queries — org_id injected from a custom JWT claim
@fraiseql.query(
    sql_source="v_user",
    inject={"org_id": "jwt:org_id"},
)
def users(limit: int = 20, offset: int = 0) -> list[User]:
    """List users in the caller's organization."""
```

{/* source: crates/fraiseql-cli/src/schema/converter/queries.rs:L20-L33 — InjectedParamSource::Jwt parsing */}

`inject` only supports `jwt:<claim>` values; compile-time validation rejects other prefixes with an error. See [Server-side injection](/features/server-side-injection) for the full reference and the SQL function signature shape.

## Role mapping

Roles are read from the configured `scope_claim` (default `"scope"`, a space-separated string) or from a dedicated `roles` claim that the IdP populates. The validator parses both shapes and surfaces them on the `SecurityContext`; field-level rules then enforce `requires_role` and `requires_scope` against the parsed values.

```json title="Example JWT payload (Auth0 namespaced claims)"
{
  "sub": "auth0|abc123",
  "email": "alice@example.com",
  "https://myapp.com/roles": ["admin", "editor"],
  "scope": "read:User write:Post",
  "iat": 1740783600,
  "exp": 1740787200,
  "iss": "https://your-tenant.auth0.com/",
  "aud": "https://api.example.com"
}
```

{/* source: crates/fraiseql-core/src/security/oidc/providers.rs:L115-L120 — scope_claim default + alternatives ("scp", "permissions") */}

```python
@fraiseql.type(requires_role="admin")
class AdminDashboard:
    total_users: int
    revenue: float
```

Field-level access control uses `fraiseql.field(requires_scope=...)`. See [Security → field-level RBAC](/features/security#field-level-rbac).

## Multi-provider setup

A single FraiseQL deployment validates one OIDC issuer at a time via `[auth]`. To support multiple providers (e.g. "log in with Google or with Auth0"), either:

- Front the deployment with a federating IdP (Auth0, Keycloak, Cognito Identity Pools) and configure that as `[auth]`. The federating IdP handles the per-upstream OAuth flows and re-issues tokens against the FraiseQL audience.
- Run one FraiseQL replica per provider behind a path-based or header-based router, each with its own `[auth]`. The router decides which replica serves the request based on the `provider` parameter or a discovery cookie.

The `/auth/v1/authorize` route (mounted when `social_login` is configured) is the framework's built-in social-login dispatcher; it accepts a `provider` parameter and the registered providers come from the social-login registry. The full surface is covered on the Auth Extensions page (to be written in a future cycle); the source of truth is `mount_auth_routes` and the `social_login` field on the server state.

{/* source: crates/fraiseql-server/src/server/routing/auth.rs:L47-L57 — social_login route mounted under /auth/v1/authorize */}

## Test the OAuth flow

After configuring a provider, verify the full authorization code exchange manually before wiring up a frontend.

<Steps>
1. **Start the login flow** — open the login URL in a browser or follow the redirect:

   ```bash
   curl -v "http://localhost:8080/auth/start"
   # Follow the Location header to the provider's authorization page
   ```

2. **Authorize in the browser** — log in and approve the consent screen. Your browser is redirected back to the callback URL with `code` and `state` query parameters.

3. **Inspect the callback response** — `GET /auth/callback?code=...&state=...` returns a `303 See Other` with:

   ```
   Set-Cookie: __Host-access_token="<jwt>"; Path=/; HttpOnly; Secure; SameSite=Strict; Max-Age=<expires_in or 300>
   ```

4. **Call a protected query** using the cookie:

   ```bash
   curl --cookie "__Host-access_token=$JWT" http://localhost:8080/graphql \
     -H "Content-Type: application/json" \
     -d '{"query":"{ me { id email } }"}'
   ```

   Or use `Authorization: Bearer $JWT` for non-browser clients.

5. **Inspect the validated session** via `[auth.me]`:

   ```bash
   curl -fsS http://localhost:8080/auth/me -H "Authorization: Bearer $JWT"
   # {"sub":"auth0|abc","user_id":"auth0|abc","expires_at":"2026-05-30T...",
   #  "email":"alice@example.com","tenant_id":"acme"}
   ```
</Steps>

{/* source: crates/fraiseql-server/src/routes/auth.rs:L432-L460 — auth_me handler returns sub, user_id, expires_at + email/display_name + extra_claims */}

## Common OAuth errors

| HTTP status | Error code | Cause | Fix |
|-------------|-----------|-------|-----|
| `400 Bad Request` | `invalid_client` | Client ID or secret wrong | Verify `client_id` matches the provider app settings; the secret comes from `${OIDC_CLIENT_SECRET}` at compile time |
| `400 Bad Request` | `redirect_uri_mismatch` | Callback URL not registered | Add `server_redirect_uri` exactly as configured to the provider's allowed redirect list |
| `400 Bad Request` | `invalid_grant` | Authorization code already used or expired | Codes are single-use and expire quickly; do not reuse or delay exchange |
| `401 Unauthorized` | `invalid_token` | Access token expired or signature invalid | Re-run the PKCE flow to obtain a fresh token |
| `401 Unauthorized` | `audience_mismatch` | Token `aud` does not match `[auth] audience` | Confirm the IdP issued the token for the configured audience identifier |
| `403 Forbidden` | `access_denied` | User denied consent or lacks required scope | Verify the scopes requested in your provider app settings |
| `500 Internal Server Error` | `token_exchange_failed` | Network error reaching the provider | Check outbound connectivity to the issuer's discovery and token endpoints |

<Aside type="tip">
OAuth errors are logged at `WARN` level. Run `RUST_LOG=debug fraiseql-server --config fraiseql.toml` to see the full error response from the provider, including its more specific `error_description`.
</Aside>

## Troubleshooting

### Redirect URI mismatch

Ensure the redirect URI matches exactly in:

1. `server_redirect_uri` in `[fraiseql.security.pkce]` or the OIDC client identity block.
2. Provider app settings (each provider has its own "Allowed Callback URLs" field).
3. The browser's actual landing URL after IdP redirect.

A trailing slash counts.

### Token validation errors

1. Check the configured `issuer` matches the `iss` claim exactly (including trailing slash for Auth0).
2. Verify the IdP's `/.well-known/openid-configuration` is reachable from the FraiseQL pod / VM.
3. Check the audience: `[auth] audience` must match the token's `aud` exactly, or be listed in `additional_audiences`.
4. Confirm the clock skew: FraiseQL tolerates up to `clock_skew_secs` (default 60, hard-capped at 300) between the FraiseQL host and the IdP.

   {/* source: crates/fraiseql-core/src/security/oidc/providers.rs:L160-L165 — MAX_CLOCK_SKEW_SECS = 300 */}

### Missing claims

1. Verify scopes requested in your provider app settings include the claims you reference in `inject={...}` and `requires_scope`.
2. Some IdPs put roles under `roles`, `groups`, `scp`, or `permissions` rather than `scope`; set `[auth] scope_claim = "..."` accordingly.
3. Azure AD uses `oid` for the persistent user identifier; `sub` differs between apps. Pin the right one for your schema.

## OAuth across transports

OIDC authentication protects all three FraiseQL transports identically. The validator runs at the routing layer before transport dispatch.

- **GraphQL and REST** — send `Authorization: Bearer <jwt>` (or the PKCE cookie) as an HTTP header.
- **gRPC** — send `authorization: bearer <jwt>` as gRPC call metadata (lowercase key, same value format).

The OAuth flow itself — authorization code exchange, token issuance, refresh — always goes through the `/auth/*` HTTP endpoints; there is no gRPC-native OAuth flow.
