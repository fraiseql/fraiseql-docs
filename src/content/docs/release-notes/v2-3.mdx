FraiseQL v2.3.0 shipped on 2026-05-25. It is the largest release since the
v2.0 line: eight new subsystems (Studio admin dashboard, WASM functions,
file Storage, WebSocket Realtime, Auth Extensions, Schema migrations CLI,
LTree Hierarchies, REST transport), sixteen Security findings (S33–S48)
plus six additional security hardenings, broad performance work on the
request hot path, and ~16 breaking changes in the Rust core driven by
error-taxonomy consolidation and lock-free reads. Two patch releases
followed: v2.3.1 (2026-05-27) fixed an axum 0.8 startup panic, and v2.3.2
(2026-05-28) unblocked `cargo publish` for the workspace.

## Summary

v2.3.0 is the platform release. Studio brings an embedded admin dashboard
served from `/studio`. Functions is a WASM trigger system covering
`after:mutation`, `before:mutation`, `after:storage`, cron, and HTTP
triggers, with WASI, AES-256-GCM secrets, and a cold-start module cache.
Storage adds an S3 / local / Azure / GCS backend abstraction with
RLS-enforced tenant isolation and file transforms (resize, watermark,
format conversion). Realtime ships a WebSocket subscription server with
RLS event delivery, a `CronScheduler`, presence rooms, and a CDC
`ObserverRuntime` tied into `EventBridge`. Auth Extensions land:
multi-provider social login, account linking by email, magic links, email
OTP, TOTP MFA with recovery codes, anonymous session signup, and phone-auth
SMS OTP. Schema migrations CLI exposes evolution support through
`fraiseql-cli`. Hierarchies adds `descendantOfId` / `ancestorOfId` LTree
operators (PostgreSQL only). REST transport gains `[rest]` TOML wiring
behind the `rest` feature flag.

The Security section consolidates the auth-path hardening sequence
(S33–S48), reaching from path-traversal guards on `reload_schema` through
PKCE, SCRAM key-material zeroization, JWT algorithm and header-injection
defences, IPv6 literal parsing, Vault rotation atomicity, and admin
bearer-token brute-force protection. Three additional hardenings tighten
multi-tenant correctness: a cache RLS isolation guard, end-to-end
WebSocket subscription tenant isolation, and RLS enforcement on the
aggregate / window query paths. Outbound HTTP from `fraiseql-functions`
now denies by default; hosts must be explicitly allowlisted.

Performance work on the request hot path includes a parsed-query AST
reused between validation and matching, an `Arc::unwrap_or_clone` response
cache hit, lock-free reads across five rate-limiter / store / index maps,
a reused scratch buffer in `compute_response_cache_key`, an
`impl Iterator` return on `extract_root_field_names`, and a
`LazyLock<Regex>` swap in the UUID extractor.

The breaking changes are dominated by error-taxonomy consolidation:
the parallel HTTP-shaped `RuntimeError` and five shadow domain enums are
deleted; `FraiseQLError` is the single workspace root, with subsystem
errors composing via owned `From<X>` impls. `FraiseQLError::Storage` is
replaced by `FraiseQLError::File(FileError::*)`. Six public DTOs receive
`#[non_exhaustive]` with `new(...)` constructors. The detailed
sixteen-row breaking-change table is below, and the full migration
walkthrough lives at
[/migrations/upgrading/v2-2-to-v2-3/](/migrations/upgrading/v2-2-to-v2-3/).

## Headline subsystems

### Studio admin dashboard

A SPA shell with embedded assets at `/studio`, backed by admin API
endpoints for schema, health, data browser, auth, storage, realtime,
functions, and metrics. The metrics endpoint
(`GET /admin/v1/metrics/summary`) is wired to the live `MetricsCollector`
with real-time latency percentiles and cache hit rate. Documented in a
forthcoming page at `/features/studio/`.

{/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L109-L110 — Studio metrics endpoint added */}

### Functions

The `fraiseql-functions` crate adds a WASM trigger system covering
`after:mutation`, `before:mutation`, `after:storage`, cron, and HTTP
triggers, dispatched through a `TriggerRegistry`. WASM host bindings cover
function execution with WASI support; host ops wire in `SqlExecutor`
injection; a sandbox and concurrency limiter cap runaway functions; and
function secrets are sealed with AES-256-GCM. A WASM module cache reduces
cold-start latency. Documented in a forthcoming page at
`/features/functions/`.

{/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L71-L77 — Functions trigger system */}

### Storage

The `fraiseql-storage` crate ships an S3 / local / Azure / GCS backend
abstraction with RLS-enforced tenant isolation, file transforms (resize,
watermark, format conversion), and access-control routes mounted on the
server. Documented in a forthcoming page at `/features/storage/`.

{/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L66-L69 — Storage API */}

### Realtime

A WebSocket server with subscription protocol, RLS-aware event delivery,
broadcast observer, `CronScheduler` for periodic tasks, presence manager
with room tracking and heartbeat eviction, broadcast channels with a REST
publish endpoint, and a CDC `ObserverRuntime` wired into `EventBridge`.
CDC filtering is tenant-aware via `fk_customer_org`. Documented in a
forthcoming page at `/features/realtime/`.

{/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L79-L84 — Realtime subsystem */}

### Auth extensions

Unified multi-provider social login (Google, GitHub, Apple, Microsoft),
account linking by email (same email → same user), magic links / email
OTP, TOTP MFA with recovery codes, anonymous session signup, and
phone-auth SMS OTP. The OIDC `/auth/me` claim-enrichment path is now
TOML-driven (programmatic configuration via `MeEnrichmentConfig` is
removed — see breaking changes). Documented in a forthcoming page at
`/features/auth-extensions/`.

{/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L89-L92 — Auth extensions added */}

### Schema migrations CLI

Schema migrations and evolution support land in `fraiseql-cli`,
exposing the schema-evolution model through CLI subcommands. Documented
in a forthcoming page at `/reference/cli/migrations/`.

{/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L100-L101 — Schema migrations CLI added */}

### Hierarchies

LTree ID-based operators `descendantOfId` and `ancestorOfId` resolve an
entity's ltree path from its UUID before performing hierarchical
comparisons. Self-referencing hierarchies (`path <@ (SELECT path FROM t
WHERE id = $1)`) and cross-table hierarchies via FK semi-joins are
supported. Configuration lives in `fraiseql.toml` under `[hierarchies]`
with `table` and `path_column` settings; a field-level `hierarchy`
annotation and compile-time validation drive the wiring. PostgreSQL only;
MySQL, SQLite, and SQL Server return `Unsupported`. Documented in a
forthcoming page at `/features/hierarchies/`.

{/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L46-L52 — LTree ID-based operators */}

### REST transport

The `[rest]` TOML section is now parsed and compiled through the full
pipeline (merger → intermediate → compiled schema). The server mounts a
read-only REST query router behind the `rest` Cargo feature flag. Based
on PR #229 by @magick93. Documented in a forthcoming page at
`/features/rest-transport/`.

  {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L201-L202 — RLS on aggregate/window paths */}

## Performance

Hot-path work concentrates on parse-once-execute-many, lock-free reads,
and removed per-request allocations.

- **Parsed-query AST reuse.** The validator no longer re-parses the query
  body; `parse_graphql_document(&str)` is exposed and
  `RequestValidator::validate_query_doc` accepts a pre-parsed
  `Document<'_, String>`. The HTTP handler parses once and feeds the same
  AST into validation and matching. [F001] (`b94abc592`)
  {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L504-L507 — parse-once hot path */}

- **Response cache hit `Arc::unwrap_or_clone`.** Cached JSON values
  return via `Arc::unwrap_or_clone` instead of a deep clone. [F002]
  (`15fd10a48`)
  {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L509-L510 — response cache hit Arc */}

- **Lock-free reads across five maps.** `InMemoryRateLimiter`,
  `TrustedDocumentStore`, `KeyedRateLimiter`, federation
  `ConnectionManager`, and observer `entity_type_index` migrate to
  `DashMap` (four) or `ArcSwap<HashMap>` (the observer index). Hot-path
  reads no longer block on a central lock. [F006, F007, F008, F013,
  F048, F056, F057] (`c5c946fb3`, `4b3e542b3`, `6f79c711e`, `3cda8124f`,
  `1ebae1f61`)
  {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L347-L367 — lock-free reads */}

- **`compute_response_cache_key` scratch buffer.** A reused
  `Vec<u8>` plus `serde_json::to_writer` removes per-argument `String`
  allocations on the cache-key path; serialization errors propagate as
  `Validation` instead of silently colliding distinct argument trees onto
  the empty-string key. [F044, F004] (`cf3a202cd`)
  {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L512-L515 — cache key scratch buffer */}

- **`extract_root_field_names` returns `impl Iterator`.** One allocation
  removed per call. [F020] (`dffa25762`)
  {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L343-L345 — impl Iterator return */}

- **`LazyLock<Regex>` in the UUID extractor.** `OnceLock<Regex>` swapped
  for `LazyLock<Regex>` in `cache/uuid_extractor.rs`. [F027] (`ccd25ee97`)
  {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L526-L527 — LazyLock swap */}

- **Federation HTTP retry source-chain preservation.**
  `execute_with_retry` threads the most recent `reqwest::Error` into
  `FraiseQLError::Internal { source }` instead of stringifying it away.
  [F025] (`500859a48`)
  {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L520-L521 — federation retry source chain */}

- **Tracing on response-cache lookups.** Structured `event =
  "hit"|"miss"|"disabled"` fields under target
  `fraiseql::cache::response`. [F040] (`ec9015e26`)
  {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L492-L493 — response-cache tracing */}

<Aside type="note" title="Why error-taxonomy consolidation">
The CHANGELOG records that the five deleted shadow domain enums
(`fraiseql_error::{AuthError, WebhookError, NotificationError,
IntegrationError, ObserverError}`) had **zero production call sites** at
removal time. The surviving subsystem error vocabularies live in their
owning crates (`fraiseql_auth`, `fraiseql_webhooks`, `fraiseql_observers`)
and compose into `FraiseQLError::{Auth, Webhook, Observer, File}` via the
sqlx-style owned `From<X>` impl pattern. `impl IntoResponse` in
`fraiseql_error::http` now wraps `FraiseQLError` directly; the umbrella
crate `fraiseql` no longer re-exports `RuntimeError`, `AuthError`, or
`WebhookError` — use `FraiseQLError` via `fraiseql::FraiseQLError` or
`fraiseql::prelude::*`.
</Aside>

## Bug fixes

The top ten fixes by impact, severity-weighted toward security,
data-integrity, and multi-tenant correctness.

1. **Server panic at startup on observer router mount** (v2.3.1; #316,
   #317). The axum 0.7 → 0.8 migration left one path-capture literal at
   the old `:listener_id` syntax (`crates/fraiseql-server/src/observers/routes.rs:128`,
   `/checkpoint/:listener_id`). axum 0.8 hard-panics at `Router::route`
   build time, so any deployment that mounted the observer changelog
   router crashed before binding the listener. The literal is now
   `{listener_id}` and the panic site is gone.
   {/* source: ~/code/fraiseql/CHANGELOG.md@frozen-SHA:L30 — axum 0.8 router panic fix */}

## Upgrade

The v2.2 → v2.3 upgrade is dominated by the error-taxonomy consolidation
(table row 1) and the `FraiseQLError::Storage` → `File(FileError::*)`
migration (table row 2). Adopters who depend only on the umbrella
`fraiseql::FraiseQLError` re-export and never matched on
`FraiseQLError::Storage { .. }` directly will see minimal disruption;
adopters with custom adapter impls, custom rate limiters spelled with the
type name in struct fields, or pattern matches on the six DTOs receiving
`#[non_exhaustive]` will work through the table above row by row.

the authoritative source records.
