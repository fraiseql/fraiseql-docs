# Migrating from FraiseQL v2.2.x to v2.3.0

> **About this document.** This is the project's first migration guide. The
> structure here — TL;DR table → "Before you start" → numbered per-change
> sections → effort table — is the convention future guides will follow at
> `docs/migration/vA.B-to-vC.D.md`. Each release section in `CHANGELOG.md`
> should link to the corresponding guide.
>
> **About v2.3.0 specifically.** v2.3.0 supersedes an abandoned release
> attempt from 2026-05-14. The orphan tag and GitHub Release that pointed
> at `a869b7667` (commit `docs: update CHANGELOG for v2.3.0 release`) were
> never published to crates.io and have been deleted. If you pinned to a
> pre-release / RC tag from that date, re-pin to the v2.3.0 official tag.

## TL;DR

The Rust core has roughly 20 breaking changes since v2.2.1, falling into
five rough categories. The two changes most adopters will spend the most
time on are the **error taxonomy consolidation** and the **`FraiseQLError::Storage` → `FraiseQLError::File`** migration; the rest are
mostly mechanical (sed) or signature-level (one extra type parameter).

| Change | Effort | Mechanical? |
|---|---|---|
| Error taxonomy consolidation (`RuntimeError` removed) | 30 min – 2 hr | Mostly (sed + a few hand-edits) |
| `FraiseQLError::Storage` → `FraiseQLError::File(FileError::*)` | 1 – 4 hr | Partially (table-driven; pattern-match rewrites by hand) |
| `ServerError::RuntimeError` → `ServerError::Engine` | 5 min | Yes (one sed) |
| `ViewName` newtype on cache-invalidation APIs | 15 min | Yes (`ViewName::from(&str)` at call sites) |
| `execute_with_projection_arc` takes `&ProjectionRequest<'_>` | 30 min | No (custom adapter impls only) |
| `KeyedRateLimiter` now generic `<C: Clock = SystemClock>` | 5 – 15 min | Sometimes (only if you name the type explicitly) |
| `extract_root_field_names` returns `impl Iterator` | 5 min | Yes (`.collect::<Vec<_>>()`) |
| `DashMap` swap on 5 maps (atomicity contracts restored in v2.3.0) | Read-only | No code change; observer reload and rate-limiter cap stay strict |
| `parking_lot::Mutex` swap (`update_heartbeat` no longer `async`) | 5 min | Yes (drop the `.await`) |
| `MetricsCollector` flattened to bare `AtomicU64` (no `Clone`) | 15 min | Sometimes |
| `ParsedQuery.source: String` → `Arc<str>` | 5 min | Sometimes (`&*parsed.source`) |
| `ValidationRule::Pattern` takes `CompiledPattern` | 10 min | Sometimes (downstream rule constructors only) |
| Workspace clippy denials + Q4 `indexing_slicing` pilot | Variable | No (only matters if you inherit `[workspace.lints]`) |
| `CompiledSchema::from_json` adds `strict_integrity: bool` | 5 – 15 min | Yes (sed pattern in section 18) |
| Public DTOs marked `#[non_exhaustive]` — use `new()` constructors | 15 – 30 min | Mostly (struct-literal → `new(...)`) |
| Schema-intermediate / OIDC `MeEnrichmentConfig` types removed | 5 min – 1 hr | No (TOML config or open issue if affected) |

If you don't have a custom storage backend or a custom `DatabaseAdapter`,
the realistic upgrade is closer to 1–2 hours of sed + recompile + a couple
of hand edits.

## Before you start

- **Rust:** workspace MSRV is the latest stable (currently 1.82+). FraiseQL
  v2.3.0 has not been tested on any earlier toolchain.
- **Process:** `cargo update -p fraiseql --precise 2.3.0` (or bump each
  sub-crate you use) → `cargo check` → fix one section at a time. Don't
  fan-out across the whole codebase before the first section compiles
  cleanly.
- **Backup branch:** make a `v2.2-baseline` tag in your service repo before
  starting. Several of these changes (DashMap atomicity, `MetricsCollector`
  `Clone`) are silent: code compiles but behaves slightly differently. A
  comparison branch is the cheapest way to catch surprises.
- **SDKs are out of scope** for this guide. If you use the Python,
  TypeScript, Go, Java, Ruby, Dart, Elixir, F#, PHP, or C# SDKs, their
  individual changelogs cover their own breaking changes. v2.3.0 ships
  with no required SDK changes — but the SDKs may version independently
  and you should read their CHANGELOGs separately.
- **CI before you push:** if your CI runs `cargo clippy --workspace
  --all-targets`, expect new warnings to surface from the v2.3.0
  workspace-level lint promotions (section 15). Either roll them up in
  the same upgrade PR or land a permissive `[lints.clippy]` override
  first.

## Migration sections

### 1. `RuntimeError` deleted; `FraiseQLError` is now the single error type

**What changed.** `fraiseql_error::RuntimeError` is gone. So are the five
shadow domain enums that lived alongside it (`fraiseql_error::AuthError`,
`fraiseql_error::WebhookError`, `fraiseql_error::NotificationError`,
`fraiseql_error::IntegrationError`, `fraiseql_error::ObserverError`).
These five had zero production call sites in any consumer; only their own
test files imported them. `FileError` survives because it has nine
real call sites (covered in section 2).

Subsystem error vocabularies — `fraiseql_auth::AuthError`,
`fraiseql_webhooks::WebhookError`, `fraiseql_observers::ObserverError` —
still exist, in their own crates, and now compose into `FraiseQLError`
via owned `From<X> for FraiseQLError` impls (the sqlx pattern: each
subsystem owns its `From` impl, keeping `fraiseql-error` a leaf crate).

The four HTTP-shape variants that used to live on `RuntimeError`
(`Internal`, `NotFound`, `RateLimited`, `ServiceUnavailable`) are now
on `FraiseQLError` directly.

**Why.** See `POLICY_DECISIONS.md` Q1. Briefly: there were no production
consumers of `RuntimeError`, the parallel hierarchy was carrying dead
weight, and "one canonical taxonomy" reduces the surface adopters
have to learn. Composed `From` impls preserve the subsystem vocabularies
where they're actually used.

**Before:**

```rust
use fraiseql::RuntimeError;

fn handle(err: RuntimeError) -> http::StatusCode {
    match err {
        RuntimeError::RateLimited { .. }       => http::StatusCode::TOO_MANY_REQUESTS,
        RuntimeError::NotFound { .. }          => http::StatusCode::NOT_FOUND,
        RuntimeError::ServiceUnavailable { .. } => http::StatusCode::SERVICE_UNAVAILABLE,
        _                                       => http::StatusCode::INTERNAL_SERVER_ERROR,
    }
}
```

**After:**

```rust
use fraiseql::FraiseQLError;

fn handle(err: FraiseQLError) -> http::StatusCode {
    match err {
        FraiseQLError::RateLimited { .. }        => http::StatusCode::TOO_MANY_REQUESTS,
        FraiseQLError::NotFound { .. }           => http::StatusCode::NOT_FOUND,
        FraiseQLError::ServiceUnavailable { .. } => http::StatusCode::SERVICE_UNAVAILABLE,
        _                                         => http::StatusCode::INTERNAL_SERVER_ERROR,
    }
}
```

**Migration steps.**

1. Build to enumerate the breaks: `cargo build 2>&1 | rg "RuntimeError"`.
2. Rename the type in your code:
   ```bash
   find . -name '*.rs' -not -path './target/*' -exec sed -i 's/\bRuntimeError\b/FraiseQLError/g' {} +
   ```
3. Drop any `use fraiseql::RuntimeError;` lines (the previous step
   left them as `use fraiseql::FraiseQLError;`, which is the right
   import).
4. Re-compile. Any remaining mismatches will be variants that *don't*
   exist on `FraiseQLError` (e.g., if you constructed an
   `AuthError`/`WebhookError` variant directly via the deleted
   `fraiseql_error::AuthError`). See section 3 for the composition
   pattern.

**Edge cases.**

- The sed in step 2 will also rewrite the literal string `"RuntimeError"`
  inside doc comments, error messages, and serialized JSON keys. Spot-check
  with `git diff` before committing; revert any string-literal hits that
  you want to preserve.
- `impl From<RuntimeError> for SomeOtherError` declarations: delete them.
  The new pattern is `impl From<FraiseQLError> for SomeOtherError`.
- If you serialized errors to JSON: the wire format is identical — the
  internal HTTP-shape variants moved between enums but kept the same
  serde-tagged shape. No client-side change.
- Commit reference: error taxonomy consolidation at `ffd3124e9`,
  `dd1c9b80f`, `230d4d238`.

### 2. `FraiseQLError::Storage` removed; storage errors are now `File(FileError::*)`

**What changed.** The catch-all `FraiseQLError::Storage { message, code: Option<String> }`
variant is gone. The 118 production call sites in `fraiseql-storage` and
`fraiseql-functions` that used to construct it have been migrated to
typed `FileError` variants under `FraiseQLError::File(FileError::*)`.
Eight new `FileError` variants cover the backend-classification space.

This is the **largest mechanical-but-not-quite-sed** change in v2.3.0,
because the `Storage { code: Some("…") }` string discriminator becomes a
typed variant per code.

**Why.** See `POLICY_DECISIONS.md` Q1. The `code` string was a
discriminator carrying business-logic meaning in an untyped field;
typed variants give you exhaustive pattern matching and preserve the
source-chain (the old `format!("backend error: {e}")` flattening lost
the underlying type, breaking `tracing`'s error-chain instrumentation
and downstream `Error::source()` walkers).

**Code-to-variant table.** Use this when migrating any code that
constructed `Storage { code: Some("…") }`:

| Old `code:` value                  | New variant                                                |
|------------------------------------|------------------------------------------------------------|
| `"permission_denied"`              | `FileError::PermissionDenied { message, source }`          |
| `"io_error"`                       | `FileError::IoError { message, source }`                   |
| `"invalid_key"`                    | `FileError::InvalidKey { message }`                        |
| `"not_implemented"`                | `FileError::NotImplemented { message }`                    |
| `"not_supported"` / `"unsupported"`| `FileError::Unsupported { message }`                       |
| `"size_limit_exceeded"`            | `FileError::SizeLimitExceeded { message, limit, actual }`  |
| `"mime_type_not_allowed"`          | `FileError::MimeTypeNotAllowed { message, mime }`          |
| `"not_found"`                      | `FileError::NotFound { id }` (pre-existing variant reused) |
| `None` (catch-all)                 | `FileError::Backend { message, source }`                   |

**Two observable HTTP-status changes.** Most callers won't notice, but
two route differently than they did under `Storage`:

1. **`FileError::NotFound` returns 404 globally** (was 400 from the
   generic `FraiseQLError::Storage` handler). This aligns the global
   status with what `fraiseql-storage`'s `storage_error_response` and
   `fraiseql-server`'s `file_error_response` already returned for
   backend not-found cases. If you had a custom error handler that
   special-cased `FraiseQLError::Storage { code: Some("not_found") }`
   to 404, you can delete it.
2. **`FileError::InvalidKey` returns 400** (was 500 under
   `Storage { code: Some("invalid_key") }`). The previous 500 was a
   bug: a caller-supplied bad key is user-fixable and 400 is the
   semantically correct status. If you had monitoring on storage-500s,
   expect a drop. If you had monitoring on storage-400s, expect an
   increase.

Every other status code is preserved: `storage_error_response` still
routes `NotFound → 404`, `PermissionDenied → 403`, and everything else
→ 500 exactly as before, but by matching on typed variants instead of
the `code` string.

**Before:**

```rust
use fraiseql::FraiseQLError;

fn convert(err: BackendError) -> FraiseQLError {
    match err.kind() {
        BackendErrorKind::PermissionDenied => FraiseQLError::Storage {
            message: format!("permission denied: {err}"),
            code:    Some("permission_denied".to_string()),
        },
        BackendErrorKind::NotFound => FraiseQLError::Storage {
            message: format!("not found: {err}"),
            code:    Some("not_found".to_string()),
        },
        _ => FraiseQLError::Storage {
            message: format!("backend error: {err}"),
            code:    None,
        },
    }
}
```

**After:**

```rust
use fraiseql::FraiseQLError;
use fraiseql::FileError;

fn convert(err: BackendError) -> FraiseQLError {
    let id = err.key().to_owned();
    match err.kind() {
        BackendErrorKind::PermissionDenied => FraiseQLError::File(FileError::PermissionDenied {
            message: err.to_string(),
            source:  Some(Box::new(err)),
        }),
        BackendErrorKind::NotFound => FraiseQLError::File(FileError::NotFound { id }),
        _ => FraiseQLError::File(FileError::Backend {
            message: err.to_string(),
            source:  Some(Box::new(err)),
        }),
    }
}
```

**Migration steps.**

1. Compile and collect mismatches: `cargo build 2>&1 | rg "FraiseQLError::Storage"`.
2. For each call site, look up the old `code` string in the table above
   and pick the new typed variant. The source chain (`#[source]` field on
   `PermissionDenied`/`IoError`/`Backend`) is new and lets you preserve
   the underlying error type — pass `Some(Box::new(original))` rather
   than stringifying.
3. Update pattern-matching code:
   ```rust
   // Before
   FraiseQLError::Storage { code: Some(c), .. } if c == "permission_denied" => { ... }
   // After
   FraiseQLError::File(FileError::PermissionDenied { .. })                  => { ... }
   ```
4. Any place that read `code` for monitoring / metrics: switch to
   `FileError::error_code()` which returns a stable `&'static str`
   (e.g. `"file_permission_denied"`). Note the prefix — the new codes
   are `file_*`, the old ones were bare.

**Edge cases.**

- `FileError::Backend` is the catch-all for the ~67 sites that previously
  used `Storage { code: None }`. If you can classify any of those more
  specifically (a known reqwest error, an sqlx error), prefer a typed
  variant + `source: Some(Box::new(e))`.
- The new variants are `#[non_exhaustive]`: construct them by field name,
  never as positional. Future variants may be added in 2.4.x patch
  releases without further breakage.
- Commit reference: F050 migration at `4c86d2e0d`, `ed80df821`,
  `aa7d59712`, `44432234f`, `acec7e435`, `76288f3ab`.

### 3. Auth / Webhook / Observer subsystem errors compose into `FraiseQLError`

**What changed.** Subsystem error types (`fraiseql_auth::AuthError`,
`fraiseql_webhooks::WebhookError`, `fraiseql_observers::ObserverError`)
are still defined in their own crates, but they now compose into
`FraiseQLError` via `From<X> for FraiseQLError` impls. The matching
variants on `FraiseQLError` are `Auth(_)`, `Webhook(_)`, `Observer(_)`,
each wrapping a `Box<dyn std::error::Error + Send + Sync>` (the
subsystem error in boxed form).

If you used to pattern-match on `RuntimeError::Auth(...)` or similar:
the variant shape on `FraiseQLError` is `Auth(Box<...>)` — to recover
the typed subsystem error, use `downcast_ref::<T>()`.

**Why.** See `POLICY_DECISIONS.md` Q1 ("Per-enum compose-vs-absorb
verdicts"). The composition pattern keeps subsystem crates owning their
own vocabulary while routing everything through a single root error
type. The `Box<dyn Error>` shape (rather than enum-of-enums) keeps the
`fraiseql-error` crate as a leaf — no reverse dependency on
`fraiseql-auth` etc.

**Before:**

```rust
// Hypothetical — RuntimeError did not actually carry subsystem variants
// pre-v2.3, but if you implemented a wrapper, this is the pattern.
match err {
    RuntimeError::Auth(auth_err) => handle_auth(auth_err),
    _                            => generic_500(),
}
```

**After:**

```rust
match err {
    FraiseQLError::Auth(boxed) => {
        // Downcast to the typed subsystem error
        if let Some(auth_err) = boxed.downcast_ref::<fraiseql_auth::AuthError>() {
            handle_auth_typed(auth_err)
        } else {
            // Some other error coerced into the Auth variant — degrade
            generic_500_with_message(boxed.to_string())
        }
    }
    FraiseQLError::Webhook(boxed)  => /* downcast to fraiseql_webhooks::WebhookError */,
    FraiseQLError::Observer(boxed) => /* downcast to fraiseql_observers::ObserverError */,
    _                              => generic_500(),
}
```

**Migration steps.**

1. Add the subsystem crates as direct dependencies in your `Cargo.toml`
   if you want to pattern-match on the typed variants. The umbrella
   `fraiseql` crate re-exports `FraiseQLError`, but not the subsystem
   error types — those live in `fraiseql_auth`, `fraiseql_webhooks`,
   `fraiseql_observers`.
2. For each pattern-match site, decide whether you need the typed error
   (downcast) or whether `Display`/`Error::source()` is enough.
3. The source chain works: `err.source()` walks `Auth(boxed) → boxed →
   underlying subsystem source` automatically. If you use `tracing`'s
   `error = ?err` instrumentation, you'll see the full chain in the
   structured output.

**Edge cases.**

- `downcast_ref::<T>()` returns `Option<&T>`. Handle the `None` case
  defensively — in v2.3.0 every `Auth` variant boxes an
  `fraiseql_auth::AuthError`, but that contract isn't enforced at the
  type level.
- Commit reference: source-chain verification tests at
  `crates/fraiseql-error/src/core_error/tests.rs` (added in `546ce43d7`).

### 4. `ServerError::RuntimeError` renamed to `ServerError::Engine`

**What changed.** The `ServerError::RuntimeError` variant in
`fraiseql-server` wraps `fraiseql_core::error::FraiseQLError` (the
engine error), not `fraiseql_error::RuntimeError` (which is now
deleted). The old name was a misnomer that confused two distinct
types. It's been renamed to `ServerError::Engine` with no semantic
change.

**Why.** The variant name pre-v2.3 was a leftover from an earlier
factoring; with `fraiseql_error::RuntimeError` deleted (section 1),
the name became actively misleading.

**Before:**

```rust
match server_err {
    ServerError::RuntimeError(e) => /* it's actually a FraiseQLError */ ...,
    _ => ...,
}
```

**After:**

```rust
match server_err {
    ServerError::Engine(e) => /* same FraiseQLError, correctly named */ ...,
    _ => ...,
}
```

**Migration steps.**

1. ```bash
   find . -name '*.rs' -not -path './target/*' -exec sed -i 's/\bServerError::RuntimeError\b/ServerError::Engine/g' {} +
   ```
2. Recompile. The `#[from]` impl on the variant is unchanged, so any
   `?`-propagation from `FraiseQLError` to `ServerError` still works.

**Edge cases.**

- The sed pattern `\bServerError::RuntimeError\b` is intentionally
  word-bounded so it won't match the deleted `fraiseql_error::RuntimeError`
  (which has no surrounding `ServerError::` prefix). Check `git diff`
  to confirm nothing else changed.
- Commit reference: `65491c2a9`.

### 5. `ViewName(Arc<str>)` newtype on cache-invalidation APIs

**What changed.** The cache-invalidation traits and helpers — `DatabaseAdapter::invalidate_views`, `DatabaseAdapter::invalidate_list_queries`,
`QueryResultCache::invalidate_views`, `QueryResultCache::invalidate_list_queries`,
`ResponseCache::invalidate_views`, `CachedDatabaseAdapter::invalidate_views`
— now take `&[ViewName]` instead of `&[String]`. Internal storage in
the cache (`accessed_views`, `view_index`, `list_index`) migrated
accordingly.

**Why.** View names are accessed on every cache write and every
invalidation. Promoting them from `String` to `Arc<str>` deduplicates
the allocation once per name across all cache references. The newtype
makes the boundary explicit so callers can't accidentally pass an
arbitrary `&str` that wasn't normalised.

**Before:**

```rust
impl DatabaseAdapter for MyAdapter {
    fn invalidate_views(&self, names: &[String]) -> Result<(), FraiseQLError> { ... }
}

// Call site
adapter.invalidate_views(&["v_user".to_string(), "v_org".to_string()])?;
```

**After:**

```rust
use fraiseql::ViewName;

impl DatabaseAdapter for MyAdapter {
    fn invalidate_views(&self, names: &[ViewName]) -> Result<(), FraiseQLError> { ... }
}

// Call site
adapter.invalidate_views(&[ViewName::from("v_user"), ViewName::from("v_org")])?;
```

**Migration steps.**

1. If you have a custom `DatabaseAdapter` impl, update the four
   `invalidate_*` method signatures.
2. At call sites, wrap the bare `&str` / `String` in `ViewName::from(...)`.
3. The `From<&str>` and `From<String>` impls on `ViewName` are both
   provided; pick whichever matches what you have.

**Edge cases.**

- `ViewName` clones cheaply (`Arc::clone` under the hood). Don't worry
  about passing it by value when it's convenient.
- Commit reference: F028, F037 at `4bf9a58b1`, `e760033ce`.

### 6. `execute_with_projection_arc` takes `&ProjectionRequest<'_>`

**What changed.** The `DatabaseAdapter::execute_with_projection_arc`
trait method used to take six positional arguments. They are now
bundled into a single borrowed struct, `&ProjectionRequest<'_>`, with
fields in the order they appear in the resulting SQL:
`SELECT … FROM … WHERE … ORDER BY … LIMIT … OFFSET`.

`ProjectionRequest` is intentionally **NOT** `#[non_exhaustive]`. A
missing field is a hard compile error — this is the deliberate design
so that future field additions force every impl to acknowledge them.

**Why.** Six positional arguments is past the threshold where a struct
becomes more readable. The borrowed-struct form also lets the adapter
peek at multiple fields without re-passing them, which several internal
adapters do for query-plan caching.

**Before:**

```rust
impl DatabaseAdapter for MyAdapter {
    async fn execute_with_projection_arc(
        &self,
        select_clause: &str,
        from_clause:   &str,
        where_clause:  Option<&str>,
        order_by:      Option<&str>,
        limit:         Option<i64>,
        offset:        Option<i64>,
    ) -> Result<...> { ... }
}
```

**After:**

```rust
use fraiseql::ProjectionRequest;

impl DatabaseAdapter for MyAdapter {
    async fn execute_with_projection_arc(
        &self,
        req: &ProjectionRequest<'_>,
    ) -> Result<...> {
        // Access via req.select_clause, req.from_clause, req.where_clause,
        // req.order_by, req.limit, req.offset
    }
}
```

**Migration steps.**

1. Update your adapter's method signature.
2. Replace argument-name reads with `req.field_name`.
3. If you construct a `ProjectionRequest` yourself (rare; usually the
   engine does this), use the struct literal — it's intentionally
   exhaustive, no `..Default::default()` allowed.

**Edge cases.**

- Most adopters don't have a custom `DatabaseAdapter` and won't hit
  this section. The built-in PostgreSQL / MySQL / SQLite / SQL Server
  adapters all migrated in the same commit.
- Commit reference: F043 at `83725aed8`.

### 7. `KeyedRateLimiter` is generic over a `Clock` trait

**What changed.** `KeyedRateLimiter` now takes a type parameter:
`KeyedRateLimiter<C: Clock = SystemClock>`. The previous boxed
`Box<dyn Fn() -> u64 + Send + Sync>` clock has been replaced with a
zero-sized `SystemClock` type. A blanket impl on
`F: Fn() -> u64 + Send + Sync` keeps closures working for tests.

The default type parameter means most callers (`KeyedRateLimiter::new(...)`
without an explicit type) compile unchanged.

**Why.** F018. The boxed-Fn dispatch was the only `Box<dyn>` on the
rate-limiter hot path; removing it both wins a small per-call
allocation and makes production limiters `Clone` (which they need to be
for `Arc<KeyedRateLimiter>::clone()` to dereference cheaply).

**Before:**

```rust
struct MyState {
    limiter: KeyedRateLimiter,
}
```

**After:**

```rust
struct MyState {
    limiter: KeyedRateLimiter,                // same — uses default <SystemClock>
    // OR, if you need to name a custom clock:
    test_limiter: KeyedRateLimiter<MockClock>,
}
```

**Migration steps.**

1. If you have struct fields typed `KeyedRateLimiter` (no params): no
   change — the default kicks in.
2. If you have struct fields typed `KeyedRateLimiter<...>`: keep the
   parameter; if your custom clock was a closure, the blanket impl
   on `Fn() -> u64` covers it.
3. In tests, prefer `KeyedRateLimiter::with_clock(MockClock::new(...))`
   over the closure form for readability — closures still work, but
   the trait form documents intent.

**Edge cases.**

- The blanket `impl<F: Fn() -> u64 + Send + Sync> Clock for F` is a
  documented **test-only** seam.  `SystemClock` is the only acceptable
  production implementation; closures carry none of its monotonicity
  guarantees and silently substituting one in production code (e.g.
  `KeyedRateLimiter::with_clock(|| 0)`) freezes the sliding window with
  no compile-time signal.  When advancing time in tests, prefer the
  `Arc<AtomicU64>`-backed monotonic pattern
  (`move || atomic.load(Ordering::Relaxed)`) used throughout
  `crates/fraiseql-auth/src/tests.rs`. Code review should reject
  `with_clock(|| ...)` outside `#[cfg(test)]` modules. The full policy
  is documented in the `Clock` trait rustdoc (F059, closed in v2.3.0).
- Commit reference: F018 at `3dca6bd67`.

### 8. `extract_root_field_names` returns `impl Iterator`

**What changed.** The helper used to return `Vec<&str>`. It now returns
`impl Iterator<Item = &str> + '_`.

**Before:**

```rust
let names: Vec<&str> = extract_root_field_names(&doc);
for n in &names { ... }
```

**After:**

```rust
let names = extract_root_field_names(&doc);          // impl Iterator
for n in names { ... }
// OR
let names: Vec<&str> = extract_root_field_names(&doc).collect();
```

**Migration steps.**

1. Drop the `Vec<&str>` annotation if you have one.
2. If you need a `Vec` (you store the result, you index it, you call
   `.len()`), append `.collect::<Vec<_>>()`.

**Edge cases.**

- The iterator borrows from `doc`. If you need owned strings, do
  `.map(|s| s.to_owned()).collect::<Vec<_>>()`.
- Commit reference: F020 at `dffa25762`.

### 9. Lock-free read paths on five concurrent maps

**What changed.** Five `Arc<Mutex<HashMap>>` / `Arc<RwLock<HashMap>>`
maps on the request hot path moved to lock-free read structures so
contention no longer serialises the request pipeline:

- `InMemoryRateLimiter` (auth crate) — `DashMap`
- `TrustedDocumentStore` — `DashMap`
- `KeyedRateLimiter` (auth crate) — `DashMap` with a strict-cap insert
  guard (see below)
- federation `ConnectionManager` — `DashMap`
- observer `entity_type_index` — `ArcSwap<HashMap>` for atomic reload

Per-key atomicity is preserved via `DashMap::entry()` everywhere the
old code held the outer lock across a read-modify-write. The two
contracts that originally weakened during the DashMap migration have
been **restored** in v2.3.0:

1. **`KeyedRateLimiter` capacity cap is enforced strictly.** New-key
   inserts run under a `parking_lot::Mutex<()>` insert guard so the
   cap-check → evict → insert sequence is atomic.  `len() <=
   max_entries` holds at every observable instant, not just after the
   burst settles.  Updates to keys already present take the lock-free
   fast path. (F057)
2. **Observer `entity_type_index` reload is snapshot-atomic.** The
   reload path rebuilds the new `HashMap` off-line and atomically
   swaps it via `ArcSwap::store(Arc::new(new_map))`.  Concurrent
   CDC-event lookups always observe either the fully-populated
   pre-reload generation or the fully-populated post-reload generation
   — never an empty or partially-rebuilt index. (F056)

The remaining three maps (`InMemoryRateLimiter`, `TrustedDocumentStore`,
federation `ConnectionManager`) use plain `DashMap` with documented
per-key best-effort atomicity; this is the same posture as in 2.2.x
for those specific maps.

`TrustedDocumentStore::resolve` / `document_count` / `replace_documents`
**dropped their `async` signature** (no remaining `.await` suspension).

**Why.** F006 / F007 / F008 / F013 / F048 originally migrated the maps
for lock-free reads; F056 and F057 then restored the two contracts
that had weakened in the process.  The combined v2.3.0 state gives
you both the lock-free read path and the strict contracts the 2.2.x
code provided.

**Migration steps.**

1. If you called `trusted_documents.resolve(...).await`, drop the
   `.await`:
   ```rust
   // Before
   let doc = store.resolve(&hash).await?;
   // After
   let doc = store.resolve(&hash)?;
   ```
2. No other call-site changes.

**Edge cases.**

- `KeyedRateLimiter::active_limiters() <= max_entries` is now strict;
  any 2.2.x integration test asserting that invariant continues to
  pass and no longer needs the `+ small_slack` relaxation we briefly
  recommended for the pre-release.
- The observer index reload window no longer produces empty-map reads;
  the snapshot swap is atomic, so concurrent CDC events under reload
  always see one of the two complete generations.
- Commit references: `c5c946fb3`, `4b3e542b3`, `6f79c711e`,
  `3cda8124f`, `1ebae1f61` (original DashMap migrations) plus F056
  / F057 closure commits in v2.3.0.

### 10. `parking_lot::Mutex` swap (`update_heartbeat` no longer `async`)

**What changed.** Two sync-only critical sections moved from
`tokio::sync::Mutex` to `parking_lot::Mutex`: `MemoryApqStorage::entries`
and `ListenerHandle::last_heartbeat`. As a knock-on,
`ListenerHandle::update_heartbeat` is no longer `async`.

Three sites that hold their lock across `.await` were intentionally
left on `tokio::sync::Mutex` and are unchanged.

**Why.** F019. `tokio::sync::Mutex` is required only when the critical
section spans an `.await`. For synchronous critical sections, `parking_lot`
is faster and doesn't require an async runtime to acquire.

**Before:**

```rust
listener.update_heartbeat().await;
```

**After:**

```rust
listener.update_heartbeat();
```

**Migration steps.**

1. Find call sites: `grep -rn "update_heartbeat()" src/`.
2. Drop the `.await`. If you propagated `?` through the result type,
   note that the function now returns `()` rather than a future-of-result
   — adjust accordingly (the previous result type was already infallible,
   so this is just removing the wrapper).

**Edge cases.**

- Commit reference: F019 at `bb95ef8e9`.

### 11. Lifecycle `tokio::spawn` collected into `JoinSet`

**What changed.** Server lifecycle spawns (SIGUSR1 handler,
usage-persistence flush, Arrow Flight gRPC server, trusted-docs reloader,
PKCE cleanup) are now collected into a per-server `tokio::task::JoinSet`
that `serve_with_shutdown` aborts and drains under the configured
shutdown timeout.

Per-request spawns (subscription event handlers, request middleware) are
not migrated — they detach as before.

**Why.** F021. Pre-v2.3, lifecycle spawns were "fire-and-forget" and
could continue running for a few hundred ms past `serve_with_shutdown`
return. The `JoinSet` gives the runtime a controlled drain.

**Migration steps.** None. This is observably more graceful shutdown
behaviour; no public API changed.

**Edge cases.**

- If you measured shutdown latency in v2.2.x by sleeping a tail of
  ~500ms, you can probably tighten that test. The drain bound is the
  configured `shutdown_timeout` (defaults to 30s but realistically
  finishes in ~ms once lifecycle tasks see the abort).
- Commit reference: F021 at `19bfd826c`.

### 12. `MetricsCollector` counters flattened; no longer `Clone`

**What changed.** 28 individual `Arc<AtomicU64>` fields on
`MetricsCollector` were replaced with plain `AtomicU64`. As a result,
`MetricsCollector` no longer derives `Clone`. Production wiring already
holds `Arc<MetricsCollector>` and clones that, so call-site syntax
(`metrics.queries_total.fetch_add(…)`) is unchanged.

**Why.** F009. The per-counter `Arc` was redundant — every reader has
already cloned the parent `Arc<MetricsCollector>`. 28 fewer allocations
on metrics-collector construction and 28 fewer atomic refcount
operations per collector clone.

**Before:**

```rust
// Holding a per-counter handle
let counter_handle: Arc<AtomicU64> = Arc::clone(&metrics.queries_total);
counter_handle.fetch_add(1, Ordering::Relaxed);
```

**After:**

```rust
// Hold the parent and borrow the counter field
let metrics: Arc<MetricsCollector> = state.metrics.clone();
metrics.queries_total.fetch_add(1, Ordering::Relaxed);
```

**Migration steps.**

1. Find call sites: `grep -rn "Arc::clone(&metrics\." src/`.
2. Replace with a clone of the parent `Arc<MetricsCollector>` (which
   you almost certainly already have), and read the counter field
   through it.

**Edge cases.**

- Commit reference: F009 at `f5ddaa59e`.

### 13. Arrow Flight multi-batch responses use `mpsc::channel(4)` backpressure

**What changed.** Four multi-batch response sites in
`crates/fraiseql-arrow/src/service.rs` converted from
`futures::stream::iter(Vec<RecordBatch>)` to a producer task feeding a
`tokio::sync::mpsc::channel(4)` wrapped in a
`tokio_stream::wrappers::ReceiverStream`. The consumer's `poll_next`
exerts backpressure on the producer.

Single-element response sites stay on `stream::iter(vec![one])`.

**Why.** F011. Pre-v2.3, a slow client receiving a multi-batch Arrow
Flight response let the server hold the full result set in memory.
With the bounded channel, the producer suspends after queueing 4
batches if the consumer hasn't drained them.

**Migration steps.** None for callers; the public stream shape is
preserved (a `Stream<Item = Result<RecordBatch, _>>`).

**Edge cases.**

- The bounded channel may slightly change ordering under simultaneous
  cancellation + drop, but you can't observe this externally.
- Commit reference: F011 at `0077a3eb1`.

### 14. `ParsedQuery.source` is now `Arc<str>`

**What changed.** `ParsedQuery.source: String` is now `Arc<str>`.
`ParsedQuery::clone()` drops its deep string copy in favour of an
atomic refcount bump. The wire form is unchanged — custom
`serialize_with` / `deserialize_with` preserves backward-compatible
JSON.

**Why.** F042. `ParsedQuery` is cloned across the cache write path on
every cache-warming insert; with hundreds of queries per second, the
String clones added up.

**Before:**

```rust
let s: &String = &parsed.source;
let owned: String = parsed.source.clone();   // deep copy
```

**After:**

```rust
let s: &str = &*parsed.source;               // deref to &str
let owned: Arc<str> = Arc::clone(&parsed.source);   // refcount bump
```

**Migration steps.**

1. If you needed `&String` (e.g., to pass to a function taking
   `&String`): change the function to take `&str` (almost always the
   right move), or coerce with `&*parsed.source`.
2. If you `.clone()`d the source: keep doing it; the `Clone` impl is
   now cheap.

**Edge cases.**

- The serde wire format is preserved, so any cached `ParsedQuery` JSON
  from v2.2.x deserializes correctly on v2.3.0.
- Commit reference: F042 at `bab30d351`.

### 15. `ValidationRule::Pattern` takes `CompiledPattern` instead of `String`

**What changed.** `ValidationRule::Pattern { pattern: String }` became
`ValidationRule::Pattern { pattern: CompiledPattern }`. Regex compilation
now happens once at construction (or at `schema.compiled.json`
deserialization) rather than on every validation call. Invalid patterns
surface at schema load instead of degrading silently per request.

**Why.** F003. Per-request regex compilation showed up in profiles.

**Before:**

```rust
ValidationRule::Pattern { pattern: r"^[a-z]+$".to_string() }
```

**After:**

```rust
use fraiseql::CompiledPattern;

ValidationRule::Pattern { pattern: CompiledPattern::try_from(r"^[a-z]+$")? }
```

**Migration steps.**

1. If you construct `ValidationRule::Pattern` directly: wrap the source
   string in `CompiledPattern::try_from(...)`. The `?` here is mandatory
   because compilation may fail at the regex-parser level — handle it.
2. If you deserialize from JSON / TOML: no change; serde handles the
   compilation automatically and surfaces invalid regexes as a
   deserialize error.

**Edge cases.**

- A `From<String>`-style helper is provided, but it `panic`s on invalid
  regex. Prefer `try_from` for adopter code.
- Commit reference: F003 at `dd4393d06`.

### 16. `QueryParam::to_sql_param` removed; use `as_sql_param_refs`

**What changed.** The `QueryParam::to_sql_param(&p) -> Box<dyn ToSql>`
helper was deleted. `QueryParam` already implemented `ToSql`, so the
boxing was redundant. The replacement helper `as_sql_param_refs(&[QueryParam])`
returns the borrowed slice form the postgres adapter actually wants.

**Before:**

```rust
let params: Vec<Box<dyn ToSql + Sync>> =
    inputs.iter().map(|p| QueryParam::to_sql_param(p)).collect();
```

**After:**

```rust
let params = fraiseql::as_sql_param_refs(&inputs);
// OR, manually:
let params: Vec<&(dyn ToSql + Sync)> =
    inputs.iter().map(|p| p as &(dyn ToSql + Sync)).collect();
```

**Migration steps.**

1. Find call sites: `grep -rn "to_sql_param" src/`.
2. Replace with `as_sql_param_refs(&...)` for the bulk case or the
   inline `iter().map(...)` form for one-off usage.

**Edge cases.**

- The new helper returns a borrow tied to the input slice's lifetime;
  hold the slice as long as you hold the result.
- Commit reference: F036 at `c9b599e15`.

### 17. Workspace clippy denials promoted; `indexing_slicing` pilot

**What changed.** The workspace-level `[lints.clippy]` table promotes
`panic`, `unreachable`, `print_stdout`, `print_stderr`, `dbg_macro`,
`todo`, `unimplemented`, `mem_forget`, `lossy_float_literal`,
`semicolon_if_nothing_returned`, `undocumented_unsafe_blocks`, and
`missing_assert_message` from `warn` to `deny`. The `nursery` and
`cargo` lint groups are also promoted to `deny`. Three crates
(`fraiseql-error`, `fraiseql-wire`, `fraiseql-storage`) additionally
deny `clippy::indexing_slicing` at the crate root as the Q4 pilot.

**Why.** Most of these lints catch genuine bugs (`panic!` in library
code, `print!` left from debugging). The Q4 pilot validates the
refactor mechanics for a workspace-wide indexing rollout planned across
v2.3.x patch releases. See `FOLLOW_UPS.md` for the per-crate rollout
order.

**Migration steps.**

1. If your service inherits the FraiseQL workspace `[lints]` table:
   you'll see new denials surface in your code. Either fix them
   per-file with a `// Reason: ...` comment + `#[allow]`, or hoist
   the allow to the function / module.
2. If your service has its own `[lints.clippy]` table that overrides
   the workspace: no change.

**Edge cases.**

- `clippy::indexing_slicing` is only denied on the three pilot crates
  in v2.3.0. The rest of the workspace is unchanged — but downstream
  code that uses any of those crates' types will see no impact (the
  lint is about how the crate's code is written, not its public API).
- Commit reference: F053 at `897a2188a`; Q4 pilot at `bb5347e82`,
  `ace13741e`, `e6567fb98`, `4d2c5d17b`, `0a829c2ff`, `04154688d`,
  `f20fc7717`, `280ff100c`, `cfe739c71`, `e514bbf25`, `4a6c94664`,
  `3c3e16089`.

### 18. `CompiledSchema::from_json` takes a `strict_integrity` second argument

**What changed.** The canonical schema-load entry point gained a
second parameter: `strict_integrity: bool`. When `true`, the loader
rejects schemas whose embedded integrity-manifest hash doesn't match
the schema body. When `false`, the loader behaves like v2.2.x (ignores
integrity drift, logs a warning).

**Why.** Schema integrity hardening landed in v2.3 as a defense-in-depth
measure against tampered or stale `schema.compiled.json` files (e.g.,
an old artifact baked into a container image). The strict mode is
opt-in so existing services don't break on rollout.

**Before:**

```rust
use fraiseql::CompiledSchema;

let schema = CompiledSchema::from_json(json_bytes)?;
```

**After:**

```rust
use fraiseql::CompiledSchema;

let schema = CompiledSchema::from_json(json_bytes, false)?;   // v2.2.x-compatible
// OR
let schema = CompiledSchema::from_json(json_bytes, true)?;    // opt into strict integrity
```

**Migration steps.**

1. Find call sites: `grep -rn "CompiledSchema::from_json" src/`.
2. Pass `false` as the second argument to preserve v2.2.x behaviour:
   ```bash
   find . -name '*.rs' -not -path './target/*' -exec sed -i \
     's/CompiledSchema::from_json(\([^)]*\))/CompiledSchema::from_json(\1, false)/g' {} +
   ```
3. Plan a follow-up to flip the flag to `true` once your schema-artifact
   pipeline (CI, container image build, etc.) reliably produces
   integrity-manifest-consistent files.

**Edge cases.**

- The sed in step 2 won't handle nested parentheses; if your call sites
  pass a complex expression like `CompiledSchema::from_json(read_file(&path)?)`,
  do the rewrite manually.
- Commit reference: `a27d8f1c5`.

### 19. Public DTOs marked `#[non_exhaustive]` — use the `new()` constructors

**What changed.** Six public DTOs received `#[non_exhaustive]` so future
field additions don't break adopters:

| Type | Crate | Constructor |
|---|---|---|
| `RelayPageResult` | `fraiseql-db` | `RelayPageResult::new(rows, total_count)` |
| `SqlProjectionHint` | `fraiseql-db` | `SqlProjectionHint::new(database, projection_template, estimated_reduction_percent)` |
| `OrderByClause` | `fraiseql-db` | `OrderByClause::new(field, direction)` |
| `ActionResult` | `fraiseql-observers` | `ActionResult::new(...)` |
| `CacheStatus` | `fraiseql-core` | (existing `new()`) |
| `EventKind` | `fraiseql-core` | (existing `new()`) |

Two of these (`RelayPageResult`, `ActionResult`) are returned by public
traits (`RelayDatabaseAdapter` and `ActionExecutor`) that downstream
implementations satisfy — those impls MUST construct via `new()`, not
struct literal.

**Why.** F043 framing inverted for these specific DTOs: most public
return-types in FraiseQL are `#[non_exhaustive]` by convention so the
library can add fields in patch releases without breaking adopters.
The six listed types were the v2.2.x outliers that hadn't received
the marker yet.

**Before:**

```rust
let result = RelayPageResult {
    rows,
    total_count: Some(42),
};
```

**After:**

```rust
let result = RelayPageResult::new(rows, Some(42));
```

**Migration steps.**

1. Find struct-literal construction sites:
   ```bash
   for t in RelayPageResult SqlProjectionHint OrderByClause ActionResult; do
     rg -F "${t} {" src/
   done
   ```
2. Replace each with the typed `new()` constructor. The new() arguments
   are in the field-declaration order; check rustdoc if uncertain.
3. Pattern-match sites need a `_` arm:
   ```rust
   match cache_status {
       CacheStatus::Hit  => ...,
       CacheStatus::Miss => ...,
       _                 => ...,   // required for #[non_exhaustive]
   }
   ```

**Edge cases.**

- If you implement a custom `RelayDatabaseAdapter` or `ActionExecutor`:
  the constructor change is mandatory; impls that produced these types
  via struct literal won't compile.
- Commit reference: `dbc9e0afc`, `e2b9944d2`, `3d8c4bce6`.

### 20. Schema-intermediate and OIDC-enrichment types removed

**What changed.** Three `pub` types from v2.2.x are gone in v2.3:

- `fraiseql_cli::schema::intermediate::operations::IntermediateSqlSourceDispatch`
- `fraiseql_core::schema::SqlSourceDispatch`
- `fraiseql_core::security::oidc::providers::MeEnrichmentConfig`

The first two belonged to a schema-compilation intermediate model that
v2.3 replaced. The third was a programmatic OIDC `/auth/me` enrichment
config that v2.3 moved to a TOML-driven path.

**Why.** Each represented a Rust-API extension point that almost no
adopter used directly, and the v2.3 rework couldn't preserve the type
shape. Removing them was cheaper than carrying the dead surface.

**Migration steps.**

- **`IntermediateSqlSourceDispatch` / `SqlSourceDispatch`:** if you
  consumed the CLI as a library to introspect schema intermediates, or
  pattern-matched on `QueryDefinition.sql_source_dispatch`, see
  `docs/architecture/compiler.md` for the v2.3 dispatch types. If your
  use case isn't covered, please open an issue describing what you
  built — the equivalent entry point may exist but be undocumented.
- **`MeEnrichmentConfig`:** move OIDC claim-enrichment configuration
  from Rust code into `fraiseql.toml`:
  ```toml
  [auth.oidc.providers.<name>.me_enrichment]
  # ... fields documented under Phase 13 in the CHANGELOG ...
  ```
  The programmatic Rust API is no longer supported.

**Edge cases.**

- If you have a fork that exported additional schema-intermediate types,
  the v2.3 dispatch refactor likely changed those too. Diff against
  `crates/fraiseql-core/src/schema/compiled/` for the new shape.

### 21. Minor signature changes

The following entries are mechanical and don't warrant their own
section; treat them as a checklist:

- **`compute_response_cache_key`** uses a reused scratch `Vec<u8>` and
  `serde_json::to_writer`. Errors propagate as `Validation` instead of
  silently colliding. No public-API change unless you implemented a
  custom cache-key strategy. [F004, F044]
- **`Response cache hit` returns `Arc::unwrap_or_clone`** of the
  cached JSON value (was a deep clone). No call-site change. [F002]
- **`UNSUPPORTED_OPERATION` API error code** now maps to HTTP 501 (Not
  Implemented) instead of 500. Adjust any monitoring that thresholded
  on 500s for this code.

## Estimated upgrade effort

These are realistic ranges, not optimistic ones. Multiply by 1.5x for
the first attempt; the second time someone in your team upgrades, the
ranges tighten.

| Adopter shape | Effort |
|---|---|
| Small — ≤ 50 FraiseQL call sites, no custom backends, no custom `DatabaseAdapter` | 1–2 hours |
| Medium — a few hundred sites, custom auth handlers, monitoring on error codes | 4–8 hours |
| Large — thousands of sites, custom storage backend, custom federation `ConnectionManager`, custom `DatabaseAdapter` | 1–2 days |
| Enterprise — all of the above plus custom metrics integrations, SDK extensions, CI clippy enforcement | 2–5 days (including CI updates) |

The bulk of the time in every tier is sections 1, 2, and 3 (error
taxonomy + Storage→File migration + Auth/Webhook/Observer downcast).
The remaining 18 sections are usually < 10 minutes each.
