Why FraiseQL for Semantic Search?

🔍 Native Vector Search

pgvector runs inside your PostgreSQL database. Single source of truth for data and vectors.

⚡ Sub-millisecond Queries

HNSW indexing with Rust-fast field selection. No latency from external vector DB calls.

🎯 Hybrid Search

Combine semantic similarity with metadata filtering. Find relevant results with precision.

💰 Cost Effective

No separate vector database costs. Everything runs on your existing PostgreSQL infrastructure.

vs External Vector Databases

❌ Traditional Approach

  • • Separate vector database (Pinecone, Weaviate)
  • • Sync data between PostgreSQL and vector DB
  • • Additional infrastructure costs
  • • Eventual consistency issues
  • • Complex deployment and monitoring
  • • Multiple query languages (SQL + vector APIs)

✅ FraiseQL + pgvector

  • • Vectors stored in PostgreSQL
  • • ACID transactions across data and vectors
  • • Single GraphQL API for everything
  • • No sync, no eventual consistency
  • • Your existing PostgreSQL infrastructure
  • • One query language: GraphQL

Implementation Example

# Semantic search GraphQL API
query SearchDocumentation($query: String!, $limit: Int) {
  documentation(
    filter: {
      contentEmbedding: {
        cosineDistance: {
          vector: $query
          threshold: 0.8
          limit: $limit
        }
      }
    }
  ) {
    id
    title
    content
    category
    similarity: _distance
  }
}
# Generated PostgreSQL query
SELECT
  id, title, content, category,
  1 - (content_embedding <=> $1) as _distance
FROM documentation
WHERE content_embedding <=> $1 <= 0.8
ORDER BY content_embedding <=> $1
LIMIT $2;
